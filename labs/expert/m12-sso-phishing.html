Objective: Maintain persistent AWS access after obtaining a stolen SSO session by creating a backdoor IAM user with programmatic access keys — establishing long-lived credentials that survive SSO token expiry.
Authorized Testing Only. This lab is for authorized testing only. Never perform these actions on systems you don't have explicit written permission to test.
Prerequisites
  • A captured SSO access token (obtained from Lab 1 or Lab 2) that grants access to an account with IAM write permissions
  • The target account must not have SCPs restricting iam:CreateUser or iam:CreateAccessKey
  • Temporary credentials obtained via aws sso get-role-credentials for a role with iam:CreateUser, iam:CreateAccessKey, and iam:AttachUserPolicy
Why not refresh_token? AWS SSO OIDC does not support the refresh_token grant type for public clients after session expiry. Attempting to use a refresh token after the parent session ends results in an InvalidGrantException. The correct persistence technique is to establish IAM credentials during the active SSO session window.
