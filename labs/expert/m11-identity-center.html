Objective: Map the entire SSO landscape — enumerate Identity Center instances, permission sets, account assignments, and IdentityStore users/groups to understand who has access to what accounts across the organization.
Authorized Testing Only. This lab is for authorized testing only. Never perform these actions on systems you don't have explicit written permission to test.
Prerequisites
  • IAM permissions: sso-admin:ListInstances, sso-admin:ListPermissionSets, sso-admin:DescribePermissionSet, sso-admin:ListAccountAssignments, sso-admin:ListAccountsForProvisionedPermissionSet, sso-admin:ListManagedPoliciesInPermissionSet
  • IdentityStore permissions: identitystore:ListUsers, identitystore:ListGroups, identitystore:ListGroupMemberships
  • These permissions are typically held by the management account or a delegated Identity Center administrator account
  • AWS CLI configured with credentials that have the above permissions
  • Optional: organizations:ListAccounts to correlate account names
1
List all Identity Center instances in the account to identify the SSO entry point. Most organizations have a single instance, but multi-instance setups exist in larger enterprises.
Terminal
$ aws sso-admin list-instances

Detection Notes
  • CloudTrail events to monitor: sso:ListAccountAssignments, sso:ListPermissionSets, sso:DescribePermissionSet, identitystore:ListUsers, identitystore:ListGroups, identitystore:ListGroupMemberships

  • Alert on enumeration patterns: ListInstances → ListPermissionSets (all) → ListAccountsForProvisionedPermissionSet (all) within a short time window

Identity Center enumeration requires sso-admin:* permissions on the management account. Delegated admin accounts may also have partial access. The IdentityStore API uses a separate endpoint and requires the IdentityStoreId, not the Instance ARN.
Objective: Discover all AWS accounts and roles accessible through an SSO session token, then map every cross-account access path available to the authenticated user.
Authorized Testing Only. This lab is for authorized testing only. Never perform these actions on systems you don't have explicit written permission to test.
Prerequisites
  • A valid AWS SSO access token — obtained via aws sso login, device code phishing (M12-L01), or theft from ~/.aws/sso/cache/
  • The SSO portal start URL for the target organization (e.g., https://myorg.awsapps.com/start)
  • No IAM credentials required — SSO portal APIs accept the bearer token directly
1
Use a captured or cached SSO access token to list all accounts the user can access. This reveals the full organizational scope visible to this identity.
Terminal
$ aws sso list-accounts --access-token "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..."
{ "accountList": [ { "accountId": "123456789012", "accountName": "production", "emailAddress": "prod@corp.com" }, { "accountId": "234567890123", "accountName": "staging", "emailAddress": "staging@corp.com" }, { "accountId": "345678901234", "accountName": "development", "emailAddress": "dev@corp.com" }, { "accountId": "456789012345", "accountName": "security-audit", "emailAddress": "security@corp.com" } ] }

Objective: Abuse misconfigured permission sets to escalate from a low-privilege SSO user to cross-account administrator access by discovering a mislabeled permission set whose name does not reflect its actual policy.
Authorized Testing Only. This lab is for authorized testing only. Never perform these actions on systems you don't have explicit written permission to test.
Prerequisites
  • A valid SSO access token for a low-privilege user assigned to the misconfigured permission set
  • Alternatively: sso-admin:ListPermissionSets + sso-admin:ListManagedPoliciesInPermissionSet to discover the misconfiguration
  • The target organization's Identity Center instance ARN
  • Lab scenario: the "DevOpsAccess" permission set has been incorrectly configured with AdministratorAccess instead of a scoped DevOps policy
1
Enumerate the policies attached to each permission set, identifying any that carry AdministratorAccess or overly broad inline policies despite having a non-admin name.
Terminal

Objective: Establish persistent access across the organization by creating a backdoor Identity Center permission set with an innocuous name but admin-level privileges, then assigning it to an attacker-controlled SSO user.
Authorized Testing Only. This lab is for authorized testing only. Never perform these actions on systems you don't have explicit written permission to test.
Prerequisites
  • IAM or SSO credentials with the following permissions: sso-admin:CreatePermissionSet, sso-admin:AttachManagedPolicyToPermissionSet, sso-admin:CreateAccountAssignment, sso-admin:ProvisionPermissionSet
  • The Identity Center instance ARN and a target account ID
  • An attacker-controlled principal (user or group) ID in the organization's IdentityStore
  • These permissions are typically only held by the management account or delegated Identity Center admin — this scenario represents post-compromise privilege abuse
