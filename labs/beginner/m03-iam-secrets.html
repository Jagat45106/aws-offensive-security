Objective: Map the IAM permissions of stolen AWS credentials using multiple enumeration tools to build a complete permission matrix.
Prerequisites:
  • AWS CLI v2 installed: aws --version
  • Python 3.8+ and pip installed
  • Tools installed: enumerate-iam (git clone https://github.com/andresriancho/enumerate-iam), pacu (pip install pacu), cloudfox (download from GitHub releases)
  • Stolen/lab AWS Access Key ID and Secret Access Key for the target identity
  • Authorized scope confirmed — all enumeration API calls are logged in CloudTrail

Objective: Use the fewest possible API calls to understand the AWS account structure, users, roles, and policies. Follows a logical recon → enumerate → identify escalation paths sequence.
Prerequisites:
  • AWS CLI v2 installed and configured with the target profile (--profile stolen)
  • Tools installed: jq
  • Target credentials configured as the stolen profile (from Lab 1, Step 1)
  • Understanding that every API call in this lab generates a CloudTrail event in the target account
1
Start with GetCallerIdentity — requires zero permissions and always works with any valid credential. This establishes the identity context for all subsequent steps.
Terminal
$ aws sts get-caller-identity --profile stolen
{ "UserId": "AIDAJQABLZS4A3QDU576Q", "Account": "123456789012", "Arn": "arn:aws:iam::123456789012:user/dev-deploy" }
# Key intel: Account ID 123456789012, IAM user "dev-deploy"
# If Arn contains ":assumed-role/" — these are temporary STS creds

Detection Risk: iam:GetAccountAuthorizationDetails is logged by CloudTrail and is a well-known reconnaissance indicator. GuardDuty may generate a Policy:IAMUser/RootCredentialUsage or Recon:IAMUser/GetPolicyVersion finding depending on the caller's identity. This call returns the complete IAM posture of the account in a single response and is considered HIGH noise. Use it deliberately, not as a routine first step.

Lateral Movement
SecureString parameters encrypted with the default aws/ssm KMS key can be decrypted by anyone with ssm:GetParameter — no explicit KMS permissions needed. This is because the default SSM key grants decrypt access to anyone in the account with SSM read access. Only parameters encrypted with a custom CMK require separate kms:Decrypt permissions.
