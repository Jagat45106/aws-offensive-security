SSO portal clone with AITM framework (M12): Host a credential-harvesting proxy that transparently relays the real AWS SSO login page on the taken-over subdomain.
Tool Limitation — Do Not Use wget for React/SPA Cloning: wget -m -k -p cannot faithfully clone single-page applications such as the AWS SSO portal, which is a React app that loads content dynamically via JavaScript. wget only captures the initial HTML shell; all dynamic routes, login flows, and API calls will break. Use an adversary-in-the-middle (AITM) proxy framework instead.
Recommended approaches for SPA cloning / AITM phishing:
  • Evilginx3 (preferred — true AITM, captures session cookies and MFA tokens in real time): Configure a phishlet for awsapps.com and point DNS to your Evilginx3 listener. Evilginx3 reverse-proxies the real SSO portal transparently.
  • Modlishka: Similar reverse-proxy AITM approach; simpler phishlet config than Evilginx3.
  • HTTrack: Suitable for static credential-capture pages only (not SPAs). Use only if you are building a simplified look-alike, not a transparent proxy.

Container escape (M14): Exploit a privileged container configuration or mounted Docker socket to escape to the underlying EKS node. See the runtime note below — use the appropriate escape path for your node's container runtime.
EKS 1.24+ uses containerd — docker.sock does not exist on these nodes. EKS migrated from Docker to containerd as the default container runtime in version 1.24 (released April 2022). On EKS 1.24+ nodes, /var/run/docker.sock will not be present. Attempting the docker.sock escape path will fail silently. Use crictl for container enumeration on containerd nodes, or use the privileged pod + nsenter escape path shown below.
Terminal

T1548 (Abuse Elevation Control Mechanism) covers UAC bypass and setuid on endpoints — it does not model cloud IAM privilege escalation. Use T1078.004 for assuming/abusing cloud IAM identities and T1098.003 for the role manipulation component.
  • Org Admin Pivot maps to T1199 (Trusted Relationship) + T1078.004 (Valid Accounts: Cloud Accounts). T1580 (Cloud Infrastructure Discovery) is an enumeration/discovery technique — it describes discovering cloud resources, not pivoting through trust relationships to gain elevated access. The cross-account trust chain pivot is an exploitation of a Trusted Relationship (T1199).
  • Cross-Account Role assumption maps to T1199 (Trusted Relationship) + T1078.004 (Valid Accounts: Cloud Accounts). T1550.001 (Use Alternate Authentication Material: Application Access Token) covers SaaS OAuth tokens and similar application-layer tokens — not AWS STS cross-account role chaining, which is a trust relationship exploitation.
